# Glossary

Терміни поділені на дві групи: **домен трейдингу** (предметна область) та **система FIX Space** (власна термінологія платформи).

---

## Домен трейдингу

**Long (довга позиція)** — позиція в розрахунку на зростання ціни активу. Прибуток виникає при підвищенні ціни після відкриття.

**Short (коротка позиція)** — позиція в розрахунку на падіння ціни активу. Прибуток виникає при зниженні ціни після відкриття.

**CFD (Contract for Difference)** — фінансовий інструмент, що дозволяє отримувати прибуток або збиток від зміни ціни базового активу без реального володіння ним.

**Дериватив (Derivative)** — фінансовий контракт, ціна якого залежить від іншого активу (базового активу). Сам дериватив не є реальним активом — лише відображає зміну його ціни.

**Базовий актив** — фінансовий інструмент, на зміну ціни якого укладається CFD-контракт (акція, нафта, золото, валютна пара). Трейдер не володіє базовим активом.

**Валютна пара** — котирування двох валют, де одна виражена через іншу. Наприклад, EUR/USD показує скільки доларів коштує 1 євро. Найпоширеніший тип базового активу у CFD-трейдингу.

**Угода (trade, позиція)** — відкриття та закриття певного об'єму контракту за певним базовим активом.

**P&L (Profit & Loss)** — фінансовий результат угоди: різниця між ціною закриття та відкриття з урахуванням обсягу, комісій та свопу.

**RR (Risk/Reward)** — співвідношення ризику до потенційного прибутку угоди. Наприклад, RR 1:2 означає ризик 1% заради прибутку 2%.

**R-Multiple** — скільки одиниць ризику (R) повернула угода. +2R означає прибуток вдвічі більший за ризик; -1R — збиток рівний ризику.

**Win Rate** — відсоток прибуткових угод від загальної кількості.

**Profit Factor** — відношення сумарного прибутку до сумарного збитку. Значення > 1 означає прибуткову систему.

**Expectancy (математичне очікування)** — середній результат на угоду з урахуванням Win Rate та середнього RR.

**Drawdown** — зниження балансу від попереднього максимуму. Ключова метрика контролю ризиків.

**Leverage (кредитне плече)** — механізм, що дозволяє відкривати позиції більшого обсягу, ніж власний баланс, використовуючи позикові кошти брокера.

**Margin (маржа)** — частина коштів трейдера, що резервується для підтримки відкритої позиції. При падінні рівня маржі нижче критичного рівня брокер виставляє маржин-кол — вимогу поповнити рахунок; якщо цього не зроблено — позиція примусово закривається (ліквідація).

**Своп (Swap)** — комісія за перенесення відкритої позиції на наступну добу. Враховується при розрахунку P&L. Важливий для трейдерів, які тримають позиції кілька днів.

**Таймфрейм** — часовий проміжок, за який формується одна свічка або бар на ціновому графіку (M1, M5, H1, D1 тощо).

**Торгова сесія** — період часу, протягом якого трейдер активно працює та виконує угоди.

**Торгова система** — набір правил для прийняття рішень: умови входу та виходу з угоди, управління ризиками, розмір позиції.

**Ризик-менеджмент** — набір правил і обмежень для контролю втрат: максимальний ризик на угоду, денний ліміт збитку тощо.

**Журнал трейдера** — сукупність записів про угоди, нотатки, помилки та аналітику. Центральний елемент роботи трейдера.

**Prop firm (Proprietary Trading Firm)** — компанія, що надає трейдеру капітал для торгівлі в обмін на частку прибутку. Встановлює жорсткі ліміти drawdown та profit target.

**Інформаційна фрагментація** — стан, при якому критично важливі дані трейдера розподілені між багатьма непов'язаними середовищами без взаємодії між собою.

**CFD-трейдер** — користувач системи, який торгує CFD та використовує платформу для організації, фіксації й аналізу своєї діяльності.

## Система FIX Space

**Робочий простір (Space)** — ізольоване середовище користувача в системі. Об'єднує всі бази даних, записи та налаштування. Кожен користувач має свій простір; може мати до 5 просторів.

**Секція (Section)** — іменована навігаційна група баз даних у сайдбарі. Не зберігає власних даних, лише організує відображення.

**База даних (Database)** — структурована колекція записів із визначеним набором властивостей. Організовує інформацію за напрямком діяльності (угоди, нотатки, помилки тощо).

**Запис (Record)** — окремий елемент бази даних. Містить набір значень властивостей (структурована частина) та контентну область (вільна частина).

**Властивість (Property)** — атрибут запису, що визначає тип і структуру даних. Формує схему бази даних. Типи: TEXT, NUMBER, DATE, CHECKBOX, DURATION, SELECT, STATUS, RELATION, FORMULA, RATING, PROGRESS.

**Контентна область (Record Content)** — вільна зона запису нижче властивостей для довільного контенту: нотатки, скріншоти, чеклісти. Структурована через рядки, колонки, блоки та компоненти. Зберігається у форматі JSON.

**Блок (Block)** — іменований контейнер компонентів для типового сценарію роботи трейдера (наприклад, «Аналіз входу», «Передсесійна рутина»). Можна зберегти до особистої бібліотеки та вставити в запис одним кліком.

**Компонент (Component)** — атомарна одиниця контенту певного типу: параграф, зображення, таблиця, чекліст, TradingView-чарт тощо. 10 типів в системі. Кожен компонент має визначений тип і набір властивостей, що описують його вміст та поведінку.

**Шаблон (Template)** — заздалегідь визначена структура запису зі встановленими значеннями за замовчуванням. Використовується для пришвидшення створення нових записів у типових сценаріях.

**Подання (View)** — збережена конфігурація відображення бази: фільтри, сортування, групування, видимість стовпців. Не впливає на дані.

**Формула (Formula)** — обчислюване поле типу FORMULA, значення якого автоматично розраховується на основі інших властивостей запису. Підтримує арифметичні операції, логічні конструкції (IF/ELSE), роботу з текстом, датами та масивами пов'язаних записів. Налаштовується через візуальний конструктор без ручного введення синтаксису.

**Автоматизація (Automation)** — правило виду «тригер → умова → дія», що виконується у фоновому режимі. Спрацьовує у момент настання відповідної події.

## Технічні терміни

**API (Application Programming Interface)** — набір правил і механізмів, за допомогою яких різні частини системи обмінюються даними. У контексті платформи — зв'язок між веб-інтерфейсом та сервером. Побудований за архітектурним стилем REST.

**JSON (JavaScript Object Notation)** — текстовий формат обміну даними, зручний як для людини, так і для обробки програмами. У системі використовується для зберігання контентної частини запису (структуроване подання блоків та компонентів).

**JWT (JSON Web Token)** — компактний URL-безпечний спосіб передачі даних між сторонами для автентифікації. У системі використовується в поєднанні з HttpOnly cookies: access token (15 хв) для авторизації запитів, refresh token зберігається в БД у вигляді хешу.

**Автентифікація** — процес перевірки особи користувача. Необхідна для того, щоб користувач мав доступ лише до свого робочого простору. Реалізована через email/пароль, JWT та HttpOnly cookies.

**ORM (Object-Relational Mapping)** — технологія, що дозволяє працювати з базою даних через об'єкти мови програмування без написання SQL вручну. У проєкті — Prisma.

**REST (Representational State Transfer)** — архітектурний стиль API, що базується на HTTP-методах (GET, POST, PATCH, PUT, DELETE) та чітко визначених ендпоінтах. Кожен ресурс має унікальний URI, а операції виконуються через стандартні методи HTTP.

**Клієнт-серверна архітектура** — модель взаємодії, при якій браузер (клієнт) надсилає запити, а сервер їх обробляє і повертає дані. Між ними існує єдина точка взаємодії — API.

**Модульний моноліт** — архітектурний підхід, де вся логіка знаходиться в одному застосунку, але розподілена по незалежних модулях із чіткими інтерфейсами між ними. Кожен домен (простори, бази даних, записи, властивості) — окремий модуль з власним контролером, сервісом і репозиторієм.

**Масштабування** — здатність системи витримувати зростання навантаження (збільшення кількості даних або користувачів) без втрати продуктивності. У проєкті забезпечується модульною архітектурою сервера та хмарним деплоєм.
