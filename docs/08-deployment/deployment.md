# Системна інфраструктура та регламент розгортання

У цьому документі наведено детальний технічний опис інфраструктурної архітектури та процесів розгортання платформи FIX Space. Системна архітектура побудована на базі гібридної хмарної моделі: клієнтський інтерфейс розгорнуто на глобальній Edge-платформі Vercel, тоді як серверна частина (API) та реляційна база даних функціонують у контейнеризованому хмарному середовищі Railway.

## Клієнтський рівень: Платформа Vercel та фреймворк Next.js

Клієнтський застосунок (`apps/web`) спроєктовано з використанням гібридних підходів до рендерингу сторінок — статичної генерації (Static Site Generation, SSG) та динамічного рендерингу на стороні сервера (Server-Side Rendering, SSR).

### Оптимізація та конфігурація процесів збирання

Процес складання проєкту в хмарі Vercel тісно інтегрований із системою Turborepo. Це дозволяє використовувати технологію віддаленого кешування (Remote Caching), що є критично важливим для монорепозиторіїв. Завдяки цьому виключається необхідність повторного збирання та тестування незмінних внутрішніх пакетів (таких як `@fixspace/domain` або спільних конфігурацій лінтера).

- **Конвеєр збирання (Pipeline)**: Vercel здійснює інсталяцію залежностей через `pnpm install` із залученням глобального кешу пакетів, після чого ініціює команду `turbo build`.
- **Середовище виконання (Runtime)**: застосунок функціонує в середовищі Node.js версії 20.x LTS.

### Конфігурація змінних середовища

Для забезпечення коректного функціонування клієнтського застосунку налаштовано такі змінні оточення:

- `NEXT_PUBLIC_API_URL` — визначає адресу точки доступу до REST API серверної частини, що використовується клієнтським браузером для здійснення мережевих запитів.

## Серверний рівень: Контейнеризована інфраструктура Railway та Docker

Серверна частина системи розгортається у вигляді ізольованих контейнерів. Це гарантує повну відповідність конфігурацій та поведінки системи в локальному середовищі розробки та у хмарі.

### Багатоетапне збирання контейнерів (Docker Multi-stage Build)

Процес створення контейнера для API-сервісу реалізований за схемою багатоетапного збирання (Multi-stage Build), що описана в оптимізованому файлі `Dockerfile`:

1. **Етап побудови (Stage 1: Builder)**: використовується повнофункціональний базовий образ `node:20`. У цьому проміжному контейнері встановлюються всі залежності (включаючи `devDependencies`), генерується клієнтський інтерфейс бази даних Prisma Client та проводиться компіляція вихідного коду TypeScript у JavaScript (`nest build`).
2. **Етап виконання (Stage 2: Runner)**: як основа використовується ультралегкий дистрибутив `node:20-alpine`. До цього контейнера копіюються лише готові скомпільовані файли (`dist`), схеми даних Prisma та мінімальний набір виробничих залежностей (`production dependencies`). Цей підхід дозволяє скоротити розмір фінального образу приблизно до 150 МБ та значно підвищує рівень безпеки, виключаючи з контейнера системні утиліти, які можуть бути потенційними векторами атак.

### Безпека та мережеві налаштування

Для захисту серверної частини від поширених загроз та оптимізації мережевого обміну реалізовано такі заходи:

- **Порти та маршрутизація**: платформа Railway автоматично призначає та передає динамічну змінну `PORT` всередину контейнера. Сервер NestJS сконфігурований для прослуховування мережевих інтерфейсів за IP-адресою `0.0.0.0` на вказаному порту.
- **Політика крос-доменних запитів (CORS)**: з метою захисту від міжсайтових атак, параметр `CORS_ORIGIN` у налаштуваннях NestJS жорстко обмежений доменним ім'ям клієнтської частини на Vercel. Дозволені методи взаємодії лімітовані списком `GET, POST, PUT, DELETE, PATCH`.
- **Криптографічний захист автентифікації**: змінна оточення `JWT_SECRET` застосовується алгоритмом HS256 для формування цифрового підпису маркерів доступу (JWT). Довжина секретного ключа становить не менше 256 біт, що запобігає можливості його компрометації шляхом прямого перебору.

## Рівень збереження даних: СКБД PostgreSQL та ORM Prisma

Як основне сховище структурованих даних застосовується керована система керування базами даних PostgreSQL версії 16.

### Інтеграція та конфігурація бази даних

Сховище даних інтегровано із сервером за допомогою сучасних практик доступу:

- **Оптимізація з'єднань (Connection Pooling)**: для раціонального використання ресурсів сервера за умов високої інтенсивності запитів підключення до бази здійснюється через механізм пулу з'єднань (Connection Pool). Ліміти активних сесій та час очікування жорстко регламентовані у конфігураційному файлі `schema.prisma`.
- **Управління міграціями**: застосування змін до структури бази даних реалізовано за допомогою команди `prisma migrate deploy`, що виконується в автоматичному режимі перед безпосереднім стартом сервера (`prestart`). Цей крок гарантує виконання лише нових скриптів міграцій без загрози втрати існуючої інформації.

## Інтеграція поштових сервісів (SMTP)

Для забезпечення роботи сервісів автентифікації (верифікація електронної пошти при реєстрації, відновлення доступу) інтегровано систему надсилання транзакційних листів на базі бібліотеки `Nodemailer` у складі модуля `MailModule`.

### Вибір сервісу та логіка роботи середовищ

У системі реалізовано динамічне підключення поштового провайдера залежно від середовища виконання:

- **Продуктове середовище (Production)**: як основний SMTP-сервіс інтегровано платформу **Resend** (хост `smtp.resend.com`, порт підключення `465` із захищеним протоколом SSL). Це забезпечує надійну доставку повідомлень без затримок.
- **Локальна розробка (Development)**: за умови відсутності конфігурації реального SMTP-хосту, сервіс автоматично ініціює тестовий потік на базі провайдера **Ethereal Email**. Усі надіслані повідомлення перехоплюються віртуальним сервером, а в консоль розробника виводиться унікальне посилання для візуального тестування верстки листа в браузері.

### Мережева конфігурація та змінні середовища

Для роботи поштового сервісу на платформі Railway налаштовано такі змінні оточення:

- `SMTP_HOST` — доменне ім'я SMTP-сервера (`smtp.resend.com`).
- `SMTP_PORT` — порт підключення (`465`).
- `SMTP_USER` — ідентифікатор авторизації (`resend`).
- `SMTP_PASS` — конфіденційний API-ключ доступу (ізольований у Railway Secrets та прихований від логів збірки).
- `MAIL_FROM` — адреса відправника, від імені якої користувачі отримують системні сповіщення (`onboarding@resend.dev`).

## Моніторинг, самодіагностика та документування

Поточний контроль працездатності системи та діагностика її стану в реальному часі здійснюються засобами спеціалізованого модуля `@nestjs/terminus`.

1. **Перевірка активності (Liveness Probe)**: уніфікований ендпоінт `/health` відповідає на запити моніторингу для підтвердження того, що сервер перебуває в робочому стані й здатний приймати вхідні з'єднання.
2. **Перевірка готовності (Readiness Probe)**: виконує комплексний аналіз внутрішніх систем, перевіряючи з'єднання з базою даних (`database`) та поточне навантаження на оперативну пам'ять процесу (`memory_heap`).
3. **Документування інтерфейсу (API Documentation)**: інтерактивне середовище Swagger UI генерується в автоматичному режимі на основі декораторів коду NestJS. Це надає актуальну специфікацію інтерфейсу прикладного програмування для розробників та зовнішніх інтеграцій.
