# Автоматизація процесів безперервної інтеграції та розгортання

Автоматизація життєвого циклу розробки програмного забезпечення (SDLC) у проєкті FIX Space реалізована за допомогою хмарної платформи GitHub Actions. Створений конвеєр (pipeline) виконує функцію контролю якості (Quality Gate), запобігаючи інтеграції нестабільного, незахищеного або невідповідного стандартам коду в основні гілки репозиторію.

## Інфраструктурний базис автоматизації

Для забезпечення стабільності та відтворюваності всіх процесів складання та тестування системи використовується такий технологічний стек:

- **GitHub Actions** — платформа для виконання автоматизованих робочих процесів (workflows) в ізольованих віртуальних середовищах на базі контейнерів `ubuntu-latest`.
- **pnpm** — інструмент керування залежностями монорепозиторію. Застосування прапорця `--frozen-lockfile` гарантує встановлення ідентичних версій пакетів на всіх етапах конвеєра, виключаючи розбіжності у збірках.
- **Node.js (версія 20 LTS)** — єдине середовище виконання для всіх завдань та скриптів автоматизації.
- **Turborepo** — інтелектуальна система керування монорепозиторієм, яка оптимізує час виконання завдань шляхом локального та віддаленого кешування результатів попередніх кроків.

## Архітектура та етапи конвеєра `ci.yml`

Центральний робочий процес автоматизації побудований за принципом послідовних каскадних перевірок. Кожне завдання (Job) використовує можливості **Turborepo** для мінімізації повторних обчислень.

### 0. Підготовка та генерація (Prepare & Generate)

Перед виконанням будь-яких перевірок система відновлює кеш та готує необхідні артефакти:

- **Кешування Turbo**: Використовується `actions/cache@v4` для збереження директорії `.turbo`. Це дозволяє "пам'ятати" результати попередніх запусків навіть без хмарного Remote Cache.
- **Генерація Prisma Client**: Виконується через `pnpm turbo db:generate`. Ця задача є кешованою: якщо схема бази даних (`schema.prisma`) не змінювалася, клієнт миттєво відновлюється з кешу.

### 1. Статичний аналіз та перевірка стилю (Linting)

Цей етап є первинною точкою входу конвеєра. Програма запускає `pnpm turbo lint`.

- **Призначення**: виявлення синтаксичних невідповідностей, потенційних логічних помилок та контроль дотримання встановлених стандартів кодування.
- **Оптимізація**: Завдяки кешуванню, перевіряються лише змінені файли. Якщо код не змінювався, етап завершується за лічені секунди (Cache Hit).

### 2. Модульне тестування та інтеграційний аналіз якості (Test & Analysis)

Виконання цього етапу розпочинається автоматично після успішного завершення стадії статичного аналізу. Запускається команда `pnpm turbo test`.

- **Інструменти тестування**: серверна частина (`apps/api`) та клієнтський інтерфейс (`apps/web`) тестуються паралельно через Turbo.
- **Аналіз покриття**: метрики покриття коду тестами генеруються автоматично та завантажуються на платформу `Codecov`.
- **Статичний аналіз у SonarCloud**: платформа виконує глибокий аналіз накопиченого технічного боргу. Перехід до наступного етапу можливий лише за умови отримання успішного статусу перевірки (Quality Gate Passed).

### 3. Аудит безпеки та перевірка вразливостей (Security Audit)

Паралельно з процесами тестування виконується автоматичний аудит сторонніх залежностей за допомогою команди `pnpm audit`.

- **Механізм контролю**: сканування дерева залежностей проєкту на наявність відомих вразливостей (CVE). Рівень чутливості аудиту налаштовано на критичний поріг `high`. Це дозволяє уникати зупинки конвеєра через некритичні попередження, водночас гарантуючи блокування збірки при виявленні високих ризиків безпеки.

### 4. Інтеграційне тестування (E2E / Integration Testing)

Цей етап вимагає розгортання повноцінного тестового оточення через Docker.

- **Контейнеризована інфраструктура**: розгортається тимчасовий контейнер PostgreSQL 16.
- **Синхронізація схеми даних**: запускається `pnpm --filter @fixspace/database exec prisma migrate deploy`.
- **Валідація бізнес-логіки**: запуск E2E тестів через `pnpm --filter @fixspace/api test:e2e`.

### 5. Збирання проєкту та генерація контейнерів (Build & Dockerization)

Після успішного проходження фаз тестування система ініціює `pnpm turbo build`:

- **Клієнтська частина**: оптимізоване збирання Next.js.
- **Серверна частина**: створення Docker-образу для API. Завдяки Turbo, якщо код API або залежних пакетів (наприклад, `domain`) не змінювався, етап Build буде пропущено завдяки кешу.

## Конфігурація середовищ та стратегія розгортання

Інфраструктура проєкту розподілена на два функціональні середовища, кожне з яких характеризується індивідуальними рівнями доступу, конфігурацією та призначенням.

### Тестове середовище Staging (Гілка `develop`)

Призначене для демонстрації поточного стану розробки зацікавленим сторонам та проведення комплексного ручного тестування перед випуском релізу.

- **Автоматизація розгортання**: процес оновлення запускається автоматично після злиття будь-якого запиту (Pull Request) із гілкою `develop`.
- **Ізоляція даних**: для роботи стейджингу використовується окрема, ізольована від продукту база даних.

### Продуктове середовище Production (Гілка `main`)

Представляє стабільну версію системи, що безпосередньо обслуговує запити кінцевих користувачів.

- **Політика безпеки**: безпосередній запис (push) у гілку `main` заблокований правилами репозиторію. Оновлення можливе виключно шляхом злиття перевіреного коду з гілки `develop` після проходження всіх кроків автоматичного контролю якості CI.
- **Безперебійне оновлення (Zero-Downtime Deployment)**: розгортання на платформі Railway виконується за допомогою Railway CLI (`railway up --ci`). Це забезпечує плавний перехід на нову версію без призупинення обслуговування користувачів.

## Стратегії оптимізації та кешування

З метою скорочення часу виконання робочих процесів у конвеєрі інтегровано трирівневу систему кешування:

1. **Кешування на рівні GitHub Actions (`cache: pnpm`)**: зберігає глобальний кеш пакетів між запусками завдань. Це скорочує час `pnpm install`.
2. **Локальне кешування Turborepo (`.turbo`)**: зберігає артефакти збирання, результати тестування та лінтингу безпосередньо в CI через `actions/cache`. Це забезпевує режим **"Full Turbo"**, де повторні запуски ідентичного коду відбуваються майже миттєво.
3. **Розумна інвалідність**: Кеш автоматично скидається при зміні `turbo.json`, файлів конфігурації ESLint, Prettier або схеми Prisma, що гарантує актуальність перевірок.
