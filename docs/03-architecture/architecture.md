# Архітектура системи

## 1. Загальна системна архітектура (High-Level Architecture)

Платформа **FIX Space** побудована на основі **дволанкової клієнт-серверної архітектури** (Client-Server Architecture) з використанням сучасного гібридного підходу до рендерингу веб-сторінок.

_Візуальна схема взаємодії високого рівня представлена на [Контекстній діаграмі системи](../diagrams/System_Context.plantuml)._

### Обґрунтування вибору клієнт-серверної моделі:

1. **Чіткий розподіл відповідальності (Separation of Concerns):**
   - **Клієнт** відповідає за візуалізацію даних, взаємодію з користувачем (Drag-and-Drop інтерфейси, графіки), локальний стан та первинну валідацію форм.
   - **Сервер** фокусується на авторизації, обчисленні бізнес-правил (формули, автоматизації), транзакційній цілісності даних та інтеграціях.
2. **Незалежне масштабування (Scalability):**
   - Статичні активи та компоненти інтерфейсу (фронтенд) розгортаються на географічно розподіленій мережі доставки вмісту (Edge CDN) від Vercel. Це забезпечує мінімальний час первинного завантаження (TTFB).
   - Логіка API та обчислень розгортається у масштабованих Docker-контейнерах Railway, кількість яких регулюється відповідно до навантаження на процесор та оперативну пам'ять.
3. **API-First концепція:**
   Бекенд розробляється як ізольований сервіс REST API. Це дає можливість у майбутньому без зміни логіки сервера підключити альтернативні клієнти (наприклад, мобільний додаток на React Native або десктопний термінал).

## 2. Структура монорепозиторію (Monorepo Layout)

Проєкт організовано як монорепозиторій під управлінням диспетчера пакетів **pnpm** (pnpm workspaces) та системи автоматизації збирання **Turborepo**.

```
fix-space/
├── apps/
│   ├── api/                    # Бекенд: NestJS додаток (REST API)
│   └── web/                    # Фронтенд: Next.js додаток (React, Tailwind CSS)
├── packages/
│   ├── database/               # Спільний пакет: Prisma schema, клієнт та міграції
│   └── domain/                 # Спільний пакет: TypeScript типи, DTO та валідаційні схеми
├── tsconfig.json               # Глобальні налаштування TypeScript
└── turbo.json                  # Конфігурація конвеєра Turborepo (build, lint, test)
```

### Переваги монорепозиторію для проєкту:

- **Єдине джерело правди для доменів (`@fixspace/domain`):** DTO (Data Transfer Objects) та правила валідації (`class-validator`) створюються один раз і використовуються як на сервері для перевірки вхідних запитів, так і на клієнті для статичного контролю типів у формах введення.
- **Спільне перевикористання коду БД (`@fixspace/database`):** Опис Prisma-схеми та згенерований клієнт знаходяться в ізольованому пакеті, гарантуючи, що бекенд та міграційні скрипти працюють з ідентичною схемою бази даних.
- **Кешування збірок (Turborepo):** Завдяки декларативному опису завдань у `turbo.json`, при зміні коду у фронтенді бекенд не перезбирається повторно (і навпаки), що скорочує час збирання та CI/CD конвеєра на 70%.

## 3. Архітектура серверної частини (NestJS API Module)

Бекенд спроєктовано за принципом **Модульного моноліту (Modular Monolith)** з використанням шаруватого шаблону проєктування (Layered Architecture).

_Взаємодія логічних контейнерів системи детально описана на [Діаграмі контейнерів](../diagrams/Container.plantuml)._

### Шари реалізації всередині кожного модуля:

Кожен модуль бекенду (наприклад, `Auth`, `Workspace`, `Database`, `Record`) розділений на три чіткі архітектурні шари:

1. **Шар представлення (Presentation Layer — Controllers):**
   - Приймає HTTP-запити.
   - Визначає маршрути (routes).
   - Забезпечує безпеку за допомогою **NestJS Guards** (перевірка JWT-токенів, ролей та власника ресурсу через `ResourceOwnerGuard`).
   - Виконує автоматичну валідацію вхідних тіл запитів (Request Bodies) за допомогою `ValidationPipe` та схем DTO.
2. **Шар бізнес-логіки (Business Logic Layer — Services/Use Cases):**
   - Виконує транзакційні алгоритми (наприклад, ініціалізація робочого простору при реєстрації `InitializeUserSpaceUseCase`).
   - Розраховує формули (Win Rate, Net PnL, RR) та виконує ланцюжки автоматизацій.
   - Є абсолютно незалежним від протоколу передачі даних (HTTP/REST) і може бути викликаний як через HTTP-контролер, так і через cron-завдання чи CLI.
3. **Шар інфраструктури та доступу до даних (Infrastructure/Data Access Layer):**
   - Взаємодіє із базою даних за допомогою `PrismaService`.
   - Надсилає листи через `MailService` (Resend SDK).
   - Завантажує файли через `StorageService` (Cloudinary SDK).

### Ізоляція доменів:

NestJS модулі мають чітку інкапсуляцію. Всі класи (сервіси), які не вказані в масиві `exports` файлу `<name>.module.ts`, приховані всередині модуля. Доступ до бази даних здійснюється через єдину точку входу — Prisma client, без прямого дублювання SQL-запитів між модулями.

_Детальна структура внутрішніх модулів бекенду та їх взаємозв'язки наведені на [Діаграмі компонентів бекенду](../diagrams/Component.plantuml)._

## 4. Архітектура клієнтської частини (Next.js Frontend)

Фронтенд розроблено на базі **Next.js 16** (App Router) з використанням гібридного підходу до рендерингу (React Server Components + Client Components).

### Рівні компонентів (Rendering Strategy):

### Клієнтське управління станом (State Management):

- **Серверний стан (Server State):** Реалізований на базі **TanStack Query (React Query)**. Він відповідає за кешування мережевих запитів, автоматичне фонове оновлення даних при перемиканні вкладок (refetch on window focus), пагінацію та оптимістичне оновлення інтерфейсу (Optimistic Updates — зміна стану UI до завершення запиту до API для покращення сприйняття інтерфейсу користувачем).
- **Глобальний стан інтерфейсу (Client State):** Реалізований через **React Context API** для легковагових глобальних даних:
  - `AuthContext` — збереження поточної сесії та профілю користувача.
  - `WorkspaceContext` — керування активним робочим простором та швидким перемиканням баз даних.
- **Локальний стан (UI State):** Хуки `useState` та `useRef` для збереження стану відкриття модальних вікон, активних вкладок та контролерів анімацій.

## 5. Архітектура збереження та безпеки даних

### Організація бази даних:

Вся інформація зберігається в реляційній базі даних **PostgreSQL 16**.

- Транзакційність (ACID) гарантує точність фінансових та математичних розрахунків трейдера (Net PnL, Win Rate).
- Всі зв'язки (1:1, 1:N, N:M) чітко типізовані на рівні Prisma-схеми. Зв'язки типу `RELATION` між пресетними базами даних підтримують каскадне видалення та оновлення для підтримання цілісності даних.
- **Connection Pooling:** Для уникнення ліміту підключень (connection limit) під час паралельних асинхронних запитів Next.js SSR та NestJS, підключення здійснюється через пул з'єднань Railway.

### Модель безпеки та доступу:

1. **Автентифікація (JWT + HttpOnly Cookies):**
   - **Access Token (JWT):** Час життя 15 хвилин. Передається в заголовку `Authorization: Bearer <token>` для авторизації запитів до API.
   - **Refresh Token:** Зберігається в базі даних (у вигляді bcrypt-хешу) та у користувача в захищеному браузерному файлі `cookie` з прапорцями `HttpOnly`, `Secure` та `SameSite: Lax`. Це робить його недоступним для XSS-атак (Cross-Site Scripting). Час життя — 30 днів.
2. **Логічна ізоляція (Multi-tenancy):**
   Всі сутності бази даних пов'язані з конкретним `userId` або `spaceId`. Запити на зміну чи читання проходять обов'язкову валідацію через `ResourceOwnerGuard` на NestJS сервісі, що повністю виключає можливість несанкціонованого доступу користувача до чужих робочих просторів чи баз даних.

## 6. Фізична топологія та розгортання (Deployment Architecture)

Розгортання системи у хмарі побудоване на базі гібридної інфраструктури:

- Клієнтська частина Next.js розміщується на Edge CDN платформі Vercel.
- Серверна частина NestJS та PostgreSQL функціонують у хмарі Railway у вигляді Docker-контейнерів.
- Медіа-активи (аватари та картинки) зберігаються у хмарі Cloudinary.

_Схема мережевого розподілу та розміщення фізичних артефактів наведена на [Діаграмі розгортання](../diagrams/Deployment.plantuml)._

## 7. Порівняльний аналіз архітектурних рішень

| Архітектурний аспект      | Вибране рішення       | Альтернатива                    | Обґрунтування вибору                                                                                                                                                                           |
| :------------------------ | :-------------------- | :------------------------------ | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Організація бекенду**   | Модульний моноліт     | Мікросервіси                    | Мікросервіси призвели б до складності розгортання, мережевих затримок (latency) та проблеми розподілених транзакцій (2PC). Модульний моноліт NestJS забезпечує ізоляцію та швидкість розробки. |
| **Взаємодія з БД**        | Prisma ORM            | TypeORM / Knex.js               | Prisma надає декларативний опис схем, автоматичну генерацію клієнта та повну статичну типізацію, що значно прискорює написання та рефакторинг коду.                                            |
| **Розгортання зображень** | Cloudinary SDK (SaaS) | Збереження на диску API-сервера | Файлова система контейнерів Railway є тимчасовою. Збереження файлів у хмарі Cloudinary забезпечує стійкість медіафайлів користувача при оновленні контейнерів.                                 |
| **Надсилання email**      | Resend API SDK        | Власний SMTP-сервер             | Більшість хмарних провайдерів блокують вихідні SMTP-порти (25, 465). Використання Resend API через HTTP гарантує високу доставку листів та простоту інтеграції.                                |
| **Клієнтський стейт**     | TanStack Query        | Redux Toolkit                   | Redux створює надлишковий шаблонний код для синхронізації з API. TanStack Query самостійно керує серверним станом, кешуванням та фоновим оновленням даних.                                     |
