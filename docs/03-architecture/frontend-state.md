# Архітектура Фронтенду та Стейт-менеджмент (FIX Space)

---

## 1. Загальна модель стейту

У FIX Space весь стейт розділяється на дві категорії з різною природою та інструментами:

| Категорія        | Що зберігає                                     | Інструмент                | Приклад                                          |
| ---------------- | ----------------------------------------------- | ------------------------- | ------------------------------------------------ |
| **Server State** | Дані з API (записи, бази, властивості, простір) | TanStack Query v5         | список Records, дані Space, профіль User         |
| **Client State** | UI-стан без відповідника на сервері             | React (useState, Context) | відкрита модалка, значення фільтра, активний таб |

Ключова ідея: **server state живе на сервері**, клієнт лише тимчасово кешує копію. Ця копія може застаріти будь-якої миті (інший пристрій, паралельна вкладка) — тому серверний стейт потребує окремого інструменту для синхронізації, а не простого `useState`.

## 2. Серверний стейт — TanStack Query v5

### Навіщо не fetch + useState?

При ручному підході розробник змушений самостійно:

- відстежувати loading/error стани
- дедублювати однакові запити, що виникають у різних компонентах
- вручну скидати дані після мутацій
- відновлювати запити при поверненні у вкладку після фокусу

TanStack Query вирішує всі ці проблеми декларативно.

### QueryClient — центральний кеш

`QueryClient` — єдина точка зберігання серверних даних у пам'яті. Створюється один раз і передається всьому дереву компонентів через `QueryClientProvider`. Всередині — це `Map<QueryKey, CachedEntry>`: кожен унікальний `queryKey` відповідає одному кешованому результату.

### Lifecycle кешованого запиту

```
Запит виконано → fresh → stale → inactive → deleted (GC)
```

| Стан         | Умова переходу                                        | Поведінка                                               |
| ------------ | ----------------------------------------------------- | ------------------------------------------------------- |
| **fresh**    | Тільки отримано                                       | Refetch не виконується                                  |
| **stale**    | `staleTime` минув (default: **0 мс**)                 | При наступному mount або фокусі вікна — фоновий refetch |
| **inactive** | Жодного підписника (компонент розмонтований)          | Дані ще в кеші, але не оновлюються                      |
| **deleted**  | `gcTime` минув (default: **5 хвилин**) після inactive | Видалено з пам'яті                                      |

При **cache hit**: дані повертаються негайно з кешу, паралельно у фоні виконується refetch — UI не блокується.

При **cache miss**: відображається стан `isPending`, виконується мережевий запит.

### Ключові параметри QueryClient

| Параметр               | Default | Призначення                                        |
| ---------------------- | ------- | -------------------------------------------------- |
| `staleTime`            | 0 мс    | Через скільки дані вважаються застарілими          |
| `gcTime`               | 5 хв    | Через скільки неактивний кеш видаляється з пам'яті |
| `refetchOnWindowFocus` | `true`  | Автоматичний refetch при поверненні у вкладку      |
| `retry`                | 3       | Кількість повторних спроб при помилці мережі       |

### useQuery — паттерн читання

```
queryKey: ['records', databaseId]     // унікальний ідентифікатор кешу
queryFn:  () => fetchRecords(databaseId)
```

Повертає: `{ data, isPending, isError, isFetching, error }`

- `isPending` — перший завантаження (немає кешу)
- `isFetching` — будь-який запит у процесі (включно з фоновим refetch)

### useMutation — паттерн запису

Після успішної мутації запускається `invalidateQueries(['records', databaseId])` — TanStack Query помічає відповідні записи кешу як stale і виконує фоновий refetch. UI оновлюється автоматично без ручного оновлення стейту.

### Конвенція query keys

Ключі будуються від загального до конкретного — це дозволяє інвалідувати цілу колекцію або конкретний елемент:

```
['spaces']                            // всі простори
['spaces', spaceId]                   // один простір
['databases', databaseId]             // одна база даних
['records', databaseId]               // всі записи бази
['records', databaseId, recordId]     // конкретний запис
['properties', databaseId]            // властивості бази
['user', 'me']                        // поточний профіль
```

## 3. Клієнтський стейт — React primitives

### useState — локальний стейт компонента

Використовується для стейту, який не потрібен за межами компонента:

- Відкрита/закрита модалка або dropdown
- Поточне значення поля вводу перед submit
- Активний таб, поточна сторінка пагінації
- Локальний toggle (показати/приховати пасворд)

### useReducer — складніші переходи

Для стейту з декількома взаємопов'язаними полями або складними правилами переходу (наприклад, багатокроковий wizard форми). Патерн `action/reducer` не потребує зовнішньої бібліотеки.

### React Context — спільний стейт для дерева

Для стейту, спільного для багатьох компонентів у піддереві без prop drilling:

- Поточний простір (Space) — передається через layout до всіх дочірніх сторінок
- Конфігурація локалі (керується `next-intl` автоматично)

### Чому немає Zustand

Для масштабу FIX Space (single-user SPA, один активний простір) складність глобального стор-менеджера не виправдана:

- Серверний стейт — покрито TanStack Query
- UI-стейт — покрито useState/useReducer/Context

Додавання Zustand було б передчасною абстракцією без реального виграшу.

## 4. Next.js App Router — Server vs Client Components

### Server Components (default)

- Рендеряться на сервері — JavaScript компонента не потрапляє до браузера
- Можуть бути `async`, виконують запити до API прямо в тілі компонента
- Немає доступу до браузерних API (localStorage, window, useState)
- Підходять для: initial page load, layouts, статичний контент, SEO

### Client Components (`'use client'`)

- Рендеряться на клієнті (hydration після SSR)
- Мають доступ до useState, useEffect, TanStack Query
- Підходять для: інтерактивний UI, форми, drag-and-drop, real-time оновлення

### Правило вибору

```
Потрібен useState / useEffect / хуки?   →  Client Component
Потрібна інтерактивність?               →  Client Component
Тільки відображення даних?              →  Server Component (default)
```

### Streaming з Suspense

Повільний запит не блокує весь рендер сторінки — `<Suspense fallback={<Skeleton />}>` дозволяє відображати решту сторінки одразу, а повільну частину — стримити коли вона буде готова.

`loading.js` — спрощений варіант Suspense для цілого сегмента маршруту.

### React.cache() — дедублювання у Server Components

Якщо кілька Server Components в одному запиті викликають однакову функцію (наприклад, `getUser()`), `React.cache` гарантує що реальний HTTP-запит виконається лише раз — результат мемоізується в межах одного серверного рендеру.

## 5. Автентифікація — стейт без стейту

Access token зберігається в **HTTP-only cookie** — JavaScript взагалі не має до нього доступу. Наслідки для архітектури фронтенду:

- Немає `authToken` у React стейті або localStorage
- Немає Context з `{ isLoggedIn, token }` — браузер автоматично відправляє cookie з кожним запитом до API
- Перевірка авторизації відбувається на рівні API (відповідь 401)
- Профіль поточного користувача — звичайний `useQuery` із ключем `['user', 'me']`

При отриманні 401 від API — redirect на `/login`. Refresh token також у HTTP-only cookie, ротація відбувається автоматично через endpoint `/auth/refresh` без участі JS.

## 6. Стратегія кешування

### TanStack Query in-memory кеш

| Аспект             | Деталь                                                                   |
| ------------------ | ------------------------------------------------------------------------ |
| Scope              | Сесія браузера — при перезавантаженні сторінки кеш скидається            |
| Інвалідація        | Явно після мутації через `queryClient.invalidateQueries`                 |
| Background refetch | При фокусі вікна (refetchOnWindowFocus) та при новому mount stale-запиту |
| Дедублювання       | Кілька компонентів з однаковим queryKey — один мережевий запит           |

### Next.js fetch memoization (Server Components)

`fetch` виклики в Server Components автоматично дедублюються React в межах одного HTTP-запиту до сервера. Між різними запитами кеш не зберігається (SSR per-request за замовчуванням).

### Що не кешується

- **Форми** — поля введення завжди у свіжому локальному стейті
- **Drag-and-drop позиції** — транзієнтний стейт під час операції

---

## 7. Drag-and-drop (@dnd-kit)

`@dnd-kit` управляє власним транзієнтним стейтом під час перетягування: який елемент активний, над якою зоною drop знаходиться вказівник. Цей стейт живе всередині бібліотеки — зовнішній стейт не потрібен.

Після завершення drop:

1. Локальний порядок оновлюється оптимістично (UI реагує одразу)
2. Виконується PATCH запит до API з новою позицією
3. `invalidateQueries` → синхронізація кешу з сервером

## 8. Таблиця рішень — що куди

| Стейт                                   | Інструмент                  | Обґрунтування                                 |
| --------------------------------------- | --------------------------- | --------------------------------------------- |
| Дані з API (Records, Properties, Space) | `useQuery` (TanStack Query) | Серверний стейт, потрібен кеш і синхронізація |
| Форма (значення полів)                  | `useState`                  | Локальний, одноразовий, не шариться           |
| Відкрита модалка / dropdown             | `useState`                  | Локальний UI без серверного відповідника      |
| Відкритий/закритий sidebar              | `useState`                  | Локальний UI                                  |
| Поточний Space у layout                 | React Context               | Спільний для всіх дочірніх компонентів        |
| Drag-and-drop операція                  | dnd-kit internal            | Транзієнтний, керується бібліотекою           |
| Auth токен                              | HTTP-only cookie            | Недоступний для JS — поза React взагалі       |
| Локаль інтерфейсу (мова)                | next-intl                   | Вбудований механізм App Router                |
| Toast / сповіщення                      | `useState` у provider       | Ephemeral UI, не потребує persistence         |
