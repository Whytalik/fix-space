# Known Issues

Цей документ фіксує функціональність, що свідомо виключена зі scope v1.0, та відомі обмеження реалізованих модулів.

---

## Not Implemented in v1.0

Перелічені нижче можливості описані у SRS (`docs/02-requirements/functional.md`), але не входять до реалізованого обсягу дипломної роботи. Причина в кожному випадку — обмеженість часового ресурсу одного розробника, а не технічна неможливість реалізації.

| Feature                                                                  | SRS розділ | Planned Version | Причина відсутності                                                                  |
| ------------------------------------------------------------------------ | ---------- | --------------- | ------------------------------------------------------------------------------------ |
| Зовнішні торгові інтеграції (Binance, Bybit, OKX, MetaTrader 5, cTrader) | §12        | v1.0            | Потребують реальних API-ключів та окремого sync-сервісу; виключено з тестового scope |
| Автоматизації (правила КОЛИ→ТО)                                          | §14        | v0.2            | Складна бізнес-логіка; покрита unit-тестами на рівні handlers, UI не реалізований    |
| Статистика та власні звіти                                               | §16        | v0.2            | Потребує агрегаційних запитів по великому обсягу даних; відкладено                   |
| Імпорт / Експорт CSV                                                     | §11        | v0.2            | Відсутні endpoint-и; не входить до Postman-колекції та test scope                    |
| Кнопка (BUTTON) як тип властивості та компонент                          | §15        | v0.2            | Тип оголошено у схемі, логіка дій не реалізована                                     |
| Онбординг (демо-простір, тур, чеклист активації)                         | §18        | v0.3            | Потребує окремого набору seed-даних і UI-компонентів                                 |
| In-app сповіщення                                                        | §19        | v0.3            | Відсутній notification-модуль у API та UI                                            |
| Кошик для видалених записів (soft delete, 30 днів)                       | §6         | v0.3            | Видалення реалізоване як hard delete; soft delete відкладено                         |
| Контентна область запису (повний редактор блоків, 19 типів компонентів)  | §7         | v0.3            | API для get/update content реалізований; UI-редактор блоків не реалізований          |
| Глобальний пошук (Ctrl+K, охоплює всі бази простору)                     | §10        | v0.2            | Пошук у межах однієї бази реалізований; глобальний — відкладено                      |
| Дублювання простору                                                      | §2         | v0.3            | Дублювання бази даних реалізоване; простір — ні                                      |
| Розширені подання (Kanban, Gallery, List)                                | §9         | v1.0+           | Реалізовано лише Table view                                                          |

## Limitations

Відомі обмеження функціональності, що реалізована в v1.0.

| Area                         | Limitation                                                                                                       | Workaround                                |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| Формули                      | Тільки backend-обчислення; UI-конфігуратор формул не реалізований — формулу можна задати лише через API          | Налаштування через Postman або seed-дані  |
| Подання (Views)              | Один тип відображення — Table; конфігурація зберігається на сервері, але UI для управління поданнями мінімальний | —                                         |
| Зміна пароля                 | Анулює всі активні сесії (всі refresh-токени видаляються)                                                        | Повторний вхід на всіх пристроях          |
| Верифікація email            | У dev-середовищі `@DevOnly()` ендпоінт дозволяє обійти верифікацію; у production недоступний                     | Тільки для розробки                       |
| Ліміт space на акаунт        | Не перевіряється на рівні API в поточній реалізації (тільки в SRS)                                               | —                                         |
| Групування записів у поданні | Конфігурація групування зберігається, але сортування всередині груп не застосовується                            | Ручне сортування через `sort` query param |
| Масове редагування записів   | Не реалізовано (тільки індивідуальний PATCH)                                                                     | Послідовні запити через Postman           |
| RELATION властивість         | Тип оголошено у схемі та handlers; логіка зв'язків між базами (cross-database) не перевіряється                  | —                                         |
| Аватар користувача           | Зберігається в локальній директорії `public/avatars/`; у production потребує заміни на object storage (S3/R2)    | Для локальної розробки функціонує         |
| Binance CSV-імпорт           | API глибини Binance обмежений 30 днями; для більшої глибини потрібен CSV — функція не реалізована                | Ручне введення даних                      |
