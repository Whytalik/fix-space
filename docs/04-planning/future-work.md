# Планована функціональність майбутніх версій

Документ описує функціональність, що свідомо виключена з v1.0 через обмеження часу та ресурсів, але архітектурно передбачена або логічно продовжує платформу. Поділ на версії відображає орієнтовний пріоритет, а не жорсткий план.

Актуальний перелік функцій v1.0 — у `../02-requirements/functional.md` та `../01-introduction/scope.md`.

---

## v2.0 — Пріоритетні розширення

Функціональність, яка безпосередньо підсилює основний сценарій використання і є найбільш очікуваною.

### Нові типи подань (Views)

| Тип          | Опис                                                                      |
| ------------ | ------------------------------------------------------------------------- |
| **Kanban**   | Стовпці — значення SELECT або STATUS; перетягування записів між стовпцями |
| **Calendar** | Записи розміщуються за DATE-властивістю; вигляди: місяць / тиждень / день |
| **Gallery**  | Картки з превью зображення або першого компонента контентної області      |
| **List**     | Компактний список без розподілу по рядках таблиці                         |
| **Timeline** | Горизонтальна вісь часу між двома DATE-властивостями                      |

Архітектурна передумова: модель `View` вже існує з полем `type`; реалізація — новий рендер на фронтенді без змін схеми.

### Двофакторна автентифікація (2FA)

- TOTP (Google Authenticator, Authy) як другий фактор при вході
- Резервні коди відновлення
- Примусове увімкнення 2FA для акаунтів з увімкненими інтеграціями (зберігають API-ключі)

### IBKR (Interactive Brokers) інтеграція

- Підключення через Flex Query API (read-only)
- Синхронізація закритих позицій за тими самими правилами, що і Binance/Bybit/OKX
- Цільова аудиторія: institutional-adjacent трейдери; виключено з v1.0 як поза межами цільової аудиторії

### Мобільний застосунок

- PWA (Progressive Web App) як перший крок: офлайн-режим через Service Worker, встановлення на домашній екран
- Нативний застосунок (React Native / Expo) як окремий проєкт: швидке додавання записів, push-сповіщення, перегляд статистики

### PDF / звітний експорт

- Генерація PDF-звіту по базі або Statistics-сторінці
- Вибір полів, часового діапазону, включення графіків
- Шаблони звітів (Trading Report, Monthly Review тощо)

## v2.1 — Екосистема та спільна робота

### Публічний API та вебхуки

- REST API з OAuth 2.0 для сторонніх застосунків
- Вебхуки: відправка HTTP-запиту при виконанні автоматизації або зміні запису
- Документація за стандартом OpenAPI 3.0
- Sandbox-середовище для розробників

### Спільні робочі простори

- Запрошення учасників за email (ролі: Viewer, Editor, Admin)
- Коментарі до записів із згадками `@`
- Журнал активності простору (хто що змінив)
- Без спільних просторів: поточна архітектура ізолює кожен `Space` до одного `ownerId`; потребує введення моделі `SpaceMember`

### Сповіщення поза застосунком

- Email-дайджест (щоденний або щотижневий огляд: нові записи, результати автоматизацій)
- Push-сповіщення (тільки для PWA / мобільного застосунку)
- Telegram / Discord бот як опціональний канал для алертів інтеграцій

### Резервне копіювання до хмари

- Підключення Google Drive або Dropbox
- Автоматичний щоденний JSON-бекап усіх баз простору
- Ручний експорт цілого простору (ZIP з CSV + JSON контент)

## v3.0 — AI та просунута аналітика

### AI-асистент

- Аналіз торгового журналу: виявлення патернів помилок, кореляцій між умовами та результатами
- Рекомендації на основі поведінкових метрик (наприклад: «Win Rate падає щочетверга — перевір торгові сесії»)
- Генерація тексту для шаблонів аналізу (на основі заповнених полів запису)

### Backtesting-інструмент

- Завантаження OHLCV-даних (CSV або через брокерну інтеграцію)
- Визначення Entry/Exit умов через формули або фільтри
- Симуляція результатів на исторических даних із відображенням у Statistics

### Розширений конструктор формул

- Глибина вкладення більше 2 рівнів
- Функції: `IF/ELSE` з кількома умовами, `SWITCH`, `COALESCE`
- Агрегати по пов'язаних записах з фільтром: `SUM(RelatedTrades where Result = "Win")`
- Посилання між базами без RELATION-властивості (lookup)

### Публічні профілі та лідербоарди (opt-in)

- Трейдер може опублікувати статистику за обраний період (анонімно або з нікнеймом)
- Порівняння метрик з агрегованими даними спільноти
- Не суперечить принципу приватності: публікація — виключно добровільна дія

## Відкладені архітектурні рішення

Питання, що потребують проєктного рішення перед реалізацією відповідної функції.

| Питання                        | Контекст                                                                                                |
| ------------------------------ | ------------------------------------------------------------------------------------------------------- |
| **Мультитенантність**          | Спільні простори потребують RBAC; поточна `ResourceOwnerGuard` перевіряє лише `ownerId`                 |
| **Real-time синхронізація**    | Колаборація потребує WebSocket або SSE; наразі архітектура — request/response                           |
| **Масштабування формул**       | При тисячах записів перерахунок ланцюжка формул може стати вузьким місцем; потрібна черга (BullMQ)      |
| **Зберігання бінарних файлів** | Зображення в контентній області зараз — без CDN; для мобільного застосунку потрібен S3-сумісний сховище |
| **Версіонування API**          | При додаванні публічного API потрібна стратегія `/v1/` → `/v2/` без breaking changes                    |
