# Цільові показники покриття коду тестами (Code Coverage Targets & Requirements)

Цей документ визначає обов'язкові цільові показники покриття коду тестами для бекенд-сервісів (NestJS API) та фронтенд-додатку (Next.js Web), які мають бути досягнуті в рамках процесу розробки та забезпечення якості системи.

---

## 1. Загальний огляд стратегії тестування

Для системи **FIX Space** передбачено впровадження комбінованого підходу до забезпечення якості:

1. **Unit-тести та компонентні тести (API & Web)**:
   - **Бекенд (NestJS API)**: Повне покриття критичної бізнес-логіки авторизації, створення користувачів, обробки помилок та безпеки.
   - **Фронтенд (Next.js Web)**: Покриття спільних UI-примітивів (`src/components/ui/primitives/*`), чистих допоміжних функцій (`src/lib/utils/*`) та глобальних кастомних хуків.
   - **Інструмент**: **Jest** + **React Testing Library** (для Web).
2. **End-to-End (E2E) UI-тести (Web)**: Мають повністю покрити всі 21 функціональні модулі системи через фреймворк **Playwright**, імітуючи реальні сценарії користувача в різних браузерах (Chromium, Firefox, WebKit).

---

## 2. Цільові метрики покриття Unit-тестами (NestJS API)

Для тестування бізнес-логіки бекенду має використовуватися Jest. Нижче наведено мінімально допустимі рівні покриття коду для успішного проходження контролю якості (Quality Gates):

### Загальні вимоги до покриття Jest API:

| Метрика                     | Мінімальний цільовий показник (%) | Опис                                                |
| :-------------------------- | :-------------------------------- | :-------------------------------------------------- |
| **Statements (Інструкції)** | >= 40.00%                         | Мінімальний відсоток виконаних інструкцій коду      |
| **Branches (Гілки)**        | >= 20.00%                         | Обов'язкове покриття логічних розгалужень (if/else) |
| **Functions (Функції)**     | >= 5.00%                          | Загальне покриття викликів функцій у системі        |
| **Lines (Рядки)**           | >= 40.00%                         | Мінімальний відсоток виконаних рядків коду          |

### Вимоги до критично важливих сервісів та модулів:

- **`auth.service.ts`** (Сервіс автентифікації): Покриття коду має складати **не менше 85%** (Statements), оскільки сервіс відповідає за безпеку, перевірку паролів та сесій.
- **`register-user.usecase.ts`** (Сценарій реєстрації): Має бути покритий тестами на **100%** (Statements) для запобігання помилкам при реєстрації нових користувачів.
- **`app-logger.service.ts`** (Логування): Цільовий показник покриття — **не менше 35%**.
- **`i18n.helper.ts`** (Локалізація): Цільовий показник покриття — **не менше 70%**.

Для повного покриття зазначеної логіки планується розробити та виконати щонайменше 30 тест-сценаріїв у тестових сюїтах (`auth.service.spec.ts`, `register-user.usecase.spec.ts`).

### 2.1. Цільові метрики покриття Jest Web (UI Primitives & Utils)

Для тестування загальних примітивів інтерфейсу та утиліт Next.js додатку використовується Jest + React Testing Library.

**Загальні вимоги до покриття Jest Web:**

| Метрика                     | Мінімальний цільовий показник (%) | Опис                                                |
| :-------------------------- | :-------------------------------- | :-------------------------------------------------- |
| **Statements (Інструкції)** | >= 50.00%                         | Відсоток покриття інструкцій у primitives та utils  |
| **Branches (Гілки)**        | >= 30.00%                         | Покриття логічних розгалужень (різні variants/size) |
| **Lines (Рядки)**           | >= 50.00%                         | Відсоток покриття рядків у primitives та utils      |

**Обов'язкові компоненти та утиліти для покриття:**

- **`Button`** (`button.tsx`): Рендеринг, варіанти, завантаження, події.
- **`Badge`** (`badge.tsx` / `StatusDot`): Візуальні варіанти та кастомні кольори.
- **`Skeleton`** (`skeleton.tsx`): Рендеринг структури та стилі анімації.
- **`src/lib/utils/`** (допоміжні функції): 100% покриття чистої математичної/форматувальної логіки.

## 3. Автоматизоване UI-тестування (Next.js Web)

Для фронтенд-додатку Next.js має бути реалізоване наскрізне тестування за допомогою **Playwright E2E** для досягнення 100% функціонального покриття користувацьких інтерфейсів.

### Вимоги до Playwright E2E:

- **Обсяг перевірки**: Має охоплювати всі 21 модуль системи (включаючи Auth, Spaces, Sections, Databases, Properties, Records, Settings, Notifications тощо).
- **Кількість E2E тест-кейсів**: Необхідно реалізувати щонайменше **21 комплексний інтеграційний сценарій** (від `TC-AUTH-028` до `TC-CUST-005`), що безпосередньо відповідають описаним User Stories.
- **Інтеграція в CI**: Всі E2E-тести мають автоматично запускатися в ізольованому середовищі GitHub Actions при спробі злиття коду в гілки `develop` та `main`.

---

## 4. Критерії приймання якості (Quality Gates)

Для забезпечення стабільності коду перед випуском релізу мають виконуватися наступні умови:

- Усі критичні функції автентифікації, реєстрації та керування сесіями повинні бути захищені модульними тестами Jest із успішним проходженням без помилок.
- Складні інтерактивні інтерфейси (таблиці баз даних, канбан-дошки, управління властивостями, Drag-and-Drop) повинні валідуватися через Playwright в реальних браузерах, щоб гарантувати стабільність клієнтської частини та відсутність регресій.
