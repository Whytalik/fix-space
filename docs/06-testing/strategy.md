# Тест-план системи FIX Space

## 1. Вступ

FIX Space — це проектована веб-платформа для організації діяльності приватного CFD-трейдера, призначена для централізації та структурування його професійного робочого простору.

**Проблема, на вирішення якої спрямована система.** Приватний трейдер накопичує велику кількість різнорідних даних: параметри угод, аналіз торгових сесій, персональні нотатки, записи про помилки та статистику. Зазвичай ці дані є розосередженими між непов'язаними інструментами (таблиці, нотатники, сторонні веб-сервіси), що призводить до інформаційної фрагментації, підвищує когнітивне навантаження та унеможливлює проведення цілісного аналізу торгової діяльності.

**Рішення.** FIX Space надає єдине централізоване робоче середовище, в якому трейдер може:

- вести торговий журнал із структурованими властивостями та вільною контентною областю в кожному записі;
- встановлювати зв'язки між записами різних баз даних (властивість типу `RELATION`);
- автоматично розраховувати торгові метрики через властивості типу `FORMULA`;
- фільтрувати, сортувати та групувати записи в реальному часі без перезавантаження сторінки;
- налаштовувати автоматизації за принципом «тригер → умова → дія» без програмування.

**Технічний стек:**

| Шар            | Технологія                                                                   |
| -------------- | ---------------------------------------------------------------------------- |
| Backend API    | NestJS 11, Node.js 20, TypeScript (модульний моноліт)                        |
| Frontend       | Next.js, React, Tailwind CSS                                                 |
| База даних     | PostgreSQL 16, Prisma ORM 7                                                  |
| Автентифікація | JWT access-токен (15 хв) + refresh-токен у БД (bcrypt-хеш, HTTP-only cookie) |
| Пошта          | Resend SDK (production) / Nodemailer (development)                           |
| Монорепо       | Turborepo + pnpm                                                             |

**Мета тестування** — забезпечення стабільності та високої технічної якості розроблюваного продукту, підтвердження повної відповідності функціональним вимогам специфікації (SRS) та виявлення дефектів до етапу промислового релізу.

**Нормативний документ:** `../requirements/functional.md`

## 4. Цілі якості (Quality Objectives)

**Primary:**

- Функціональна відповідність SRS для всіх модулів IN SCOPE
- Коректна обробка помилок: 400 (невалідні дані), 401 (не автентифіковано), 403 (немає прав), 404 (не знайдено), 429 (rate limit)
- Коректна перевірка власності ресурсів через `ResourceOwnerGuard`

**Secondary:**

- Відсутність Critical та High дефектів на момент захисту
- Statement coverage ≥ 70% на критичних сервісах бізнес-логіки

### 4.1 Метрики якості (на основі ISO/IEC 25010)

## 5. Підхід до тестування (Test Approach)

- **Analytical** — аналіз вимог SRS → визначення тест-сценаріїв → написання тест-кейсів
- **Requirements-based** — кожна функціональна вимога SRS має бути прив'язана до відповідного тест-сценарію через матрицю RTM
- **Risk-based** — встановлюється найвищий пріоритет тестування для критичних шляхів системи: Auth flow та Record CRUD
- **Continuous Quality Control** — тестування має проводитися паралельно з кожною ітерацією розробки відповідно до DoD

## 6. Типи тестування

### 6.1 Модульне тестування (Unit Testing)

| Атрибут              | Значення                                                          |
| -------------------- | ----------------------------------------------------------------- |
| Метод                | White Box                                                         |
| Інструмент           | Jest 29 + ts-jest                                                 |
| Покриття             | 44 spec-файли, 481+ тест-кейси                                    |
| Мокування            | `jest.fn()`, NestJS `TestingModule`                               |
| Запуск               | `pnpm --filter @fixspace/api test`                                |
| Техніки тест-дизайну | Equivalence Partitioning, Boundary Value Analysis, Error Guessing |

Тести перевіряють ізольовану логіку сервісів, handlers та утиліт без реального підключення до БД.

### 6.2 Інтеграційне тестування API

| Атрибут      | Значення                                                                |
| ------------ | ----------------------------------------------------------------------- |
| Метод        | Grey Box                                                                |
| Інструмент   | Jest + Supertest                                                        |
| Покриття     | 18 spec-файлів, 155 тест-кейсів                                         |
| Середовище   | Ізольована тестова БД `FIX Space_test` (контейнер `db-test`, порт 5433) |
| Конфігурація | `.env.test` (окремий файл, `NODE_ENV=test` встановлюється автоматично)  |
| Запуск       | `pnpm --filter @fixspace/api test:integration`                          |
| Flow         | `register → verify → login → refresh → logout → forgot/reset password`  |

Тести перевіряють HTTP-контракти ендпоінтів з реальною базою даних (Prisma). На відміну від unit-тестів, інтеграційні тести не мокають БД — вони валідують, що запит → сервіс → Prisma → PostgreSQL працюють як єдине ціле. Мокаються тільки зовнішні сервіси (пошта, ініціалізація простору).

Інтеграційні тести швидші за E2E (не потрібен браузер) і дають швидкий зворотний зв'язок при змінах в API контрактах.

> [!NOTE]
> **Чому це не E2E:**
> Справжнє E2E тестування виконується через браузер за допомогою Playwright (§6.6). Інтеграційні тести (§6.2) перевіряють API контракти на рівні HTTP → БД, але не перевіряють UI, рендеринг, редіректи та міжсторінкову навігацію.

### 6.3 Функціональне тестування — Black Box API

| Атрибут              | Значення                                          |
| -------------------- | ------------------------------------------------- |
| Метод                | Black Box                                         |
| Інструмент           | Postman v10                                       |
| Покриття             | 62 ендпоінти, усі модулі IN SCOPE                 |
| Колекція             | `docs/06-testing/postman/postman_collection.json` |
| Техніки тест-дизайну | EP, BVA, State Transition, Decision Tables        |

Тести виконуються вручну. Токен автоматично зберігається після login через Postman test script.

### 6.5 Frontend Component Testing

| Атрибут              | Значення                                                      |
| -------------------- | ------------------------------------------------------------- |
| Метод                | Black Box (рендер компонентів)                                |
| Інструмент           | Jest 29 + React Testing Library + jsdom                       |
| Конфігурація         | `next/jest` wrapper — автоматичний SWC transform, CSS mocking |
| Покриття             | UI primitives, utility functions, custom hooks                |
| Запуск               | `pnpm --filter @fixspace/web test`                            |
| Техніки тест-дизайну | Rendering verification, props validation, event handling      |

Тести перевіряють ізольований рендер React-компонентів у jsdom-середовищі. `next/jest` автоматично налаштовує трансформ через Next.js Compiler, мокає CSS/image imports та завантажує `.env` файли.

**Область застосування Jest на фронтенді (Next.js Web):**

- **Включено (IN SCOPE):** Спільні UI-примітиви дизайну (`src/components/ui/primitives/*`), чисті допоміжні функції (`src/lib/utils/*`) та глобальні кастомні хуки.
- **Виключено (OUT OF SCOPE):** Сторінки (`page.tsx`), макети (`layout.tsx`) та великі форми функціональних фіч (наприклад, `LoginForm`, `RegisterForm`). Вони мають занадто багато зовнішніх залежностей (Next.js router, локалізація, сесії), тому їх тестування повністю перенесене на Playwright E2E.

**Що тестується:**

| Рівень            | Приклад                                              | Пріоритет |
| ----------------- | ---------------------------------------------------- | --------- |
| UI primitives     | `Skeleton`, `Button`, `Badge` — рендер, класи, props | High      |
| Utility functions | Чисті функції з `src/lib/utils/`                     | High      |
| Custom hooks      | Context hooks (theme, locale, auth)                  | Medium    |

**Приклад тесту компонента:**

```tsx
import { render } from "@testing-library/react";
import { Skeleton } from "@/components/ui/primitives/skeleton";

describe("Skeleton", () => {
  it("renders a div element", () => {
    const { container } = render(<Skeleton />);
    expect(container.firstChild).toBeInTheDocument();
  });

  it("applies default classes", () => {
    const { container } = render(<Skeleton />);
    const div = container.firstChild as HTMLElement;
    expect(div).toHaveClass("animate-pulse");
  });
});
```

## 7. Піраміда тестування FIX Space

FIX Space використовує **класичну піраміду тестування** (Mike Cohn Test Pyramid). Основа — широке модульне покриття кожного сервісу та утиліти, поверх якого будуються інтеграційні тести ключових HTTP-контрактів, і лише найкритичніші наскрізні сценарії покриваються E2E у браузері. Така форма забезпечує максимально швидкий зворотний зв'язок: unit-тести запускаються за секунди і покривають кожен модуль ізольовано; integration-тести перевіряють, що HTTP → сервіс → БД працюють разом для пріоритетних шляхів; E2E-тести підтверджують досяжність цілі реального користувача у браузері.

```
              ╭────────────────────╮
              │  E2E (Playwright)  │  ← 10%: ключові user journeys
              │        10%         │
            ╭─┴────────────────────┴─╮
            │  Integration (Jest +   │  ← 20%: HTTP → сервіс → реальна БД
            │    Supertest)  20%     │
          ╭─┴────────────────────────┴─╮
          │    Unit (Jest)    70%       │  ← 70%: повне покриття всіх сервісів
          ╰────────────────────────────╯
```

### Цільовий розподіл та відсотки тестування:

| Рівень тестування              | Вага (%)       | Область застосування у FIX Space                                                                                            | Інструменти        |
| :----------------------------- | :------------- | :-------------------------------------------------------------------------------------------------------------------------- | :----------------- |
| **Static Analysis**            | — (миттєво)    | Типи, синтаксис, відповідність стандартам стилю коду.                                                                       | TypeScript, ESLint |
| **Unit (Модульне)**            | **70%**        | Повне покриття всіх сервісів, use cases, утиліт, DTO-валідації, формул, парсерів — кожен модуль ізольовано без реальної БД. | Jest               |
| **Integration (Інтеграційне)** | **20%**        | HTTP-контракти ключових модулів (auth, databases, records, statistics) з реальною БД; перевірка guards та pipes.            | Jest + Supertest   |
| **E2E (Клієнт-серверне)**      | **10%**        | Критичні user journeys у браузері: авторизація, створення запису, перегляд статистики.                                      | Playwright         |
| **Manual (Ручне)**             | — (за потреби) | Ручна перевірка нових або рідкісних ендпоінтів; smoke-тестування після деплою.                                              | Postman            |

**Правила вибору рівня тестування:**

1. Якщо можна перевірити TypeScript або ESLint — не пиши тест.
2. Якщо логіка ізольована (немає HTTP/БД/файлової системи) — unit test.
3. Якщо перевіряєш HTTP → сервіс → БД — integration test (Supertest).
4. Якщо перевіряєш, чи може користувач досягти цілі через UI — E2E (Playwright).
5. Якщо сценарій рідкісний або разовий — Postman.

- `unit.md` — модульні тести бізнес-сервісів (White Box);

## 9. Entry / Exit Criteria

### Entry Criteria (умови початку тестування)

- API запущено і відповідає на вхідні запити
- Dev PostgreSQL доступна, всі міграції застосовані (`turbo db:migrate:dev`)
- Файл `.env.development` сконфігуровано згідно з `.env.example`
- Для Integration тестів: тестовий контейнер `db-test` запущено (`docker-compose up -d db-test`), файл `.env.test` існує, міграції на `FIX Space_test` застосовані (`cross-env NODE_ENV=test turbo db:migrate:dev`)
- Smoke test пройдено: `POST /auth/login` з валідними даними повертає `200`
- Код пройшов self-review (дипломний проект, один розробник)

### Exit Criteria (умови завершення тестування)

- 0 відкритих дефектів із рівнем Severity: Critical або High
- Покриття критичного коду (Statement coverage) повинно складати не менше 70% (підтверджено звітом про покриття)
- Усі тест-кейси рівня P1 повинні мати статус Pass
- Запланована Postman-колекція виконана без Critical помилок

## 10. Критерії зупинки та відновлення тестування

### Зупинка тестування

- Critical дефект блокує більше 50% функціональності (наприклад, auth-модуль повністю недоступний)
- База даних недоступна або пошкоджена
- API не запускається через помилку конфігурації або збірки

### Відновлення тестування

- Дефект виправлено та підтверджено повторним тестом (re-test)
- Середовище відновлено: БД скинута (`pnpm db:reset`), API перезапущено
- Тестування відновлюється з останнього стабільного checkpoint

## 11. Bug Lifecycle

```
New → Assigned → In Progress → Fixed → Re-test → Closed
                                      ↓
                               Reopened (якщо не виправлено)
```

**Додаткові статуси:** `Duplicate`, `Won't Fix`, `Not a Bug`, `Deferred`

## 12. Severity та Priority

### Severity (серйозність дефекту)

| Рівень   | Опис                                             | Приклад у контексті FIX Space                                  |
| -------- | ------------------------------------------------ | -------------------------------------------------------------- |
| Critical | Система не запускається або втрачаються дані     | `POST /auth/login` не працює; запис зникає після збереження    |
| High     | Ключова функція недоступна, workaround відсутній | `POST /spaces` повертає 500; фільтрація записів не працює      |
| Medium   | Функція працює некоректно, але є workaround      | Пагінація ігнорує `pageSize`; дублювання не копіює властивості |
| Low      | Незначна проблема, не впливає на функціонал      | Некоректне повідомлення помилки; зайвий пробіл у відповіді     |

### Priority (пріоритет виправлення)

| Рівень                | Опис                                               |
| --------------------- | -------------------------------------------------- |
| P1 Must Fix           | Виправити негайно — продукт не може бути випущений |
| P2 Should Fix         | Виправити якомога швидше, в поточній ітерації      |
| P3 Fix When Have Time | Виправити в межах поточного релізу                 |
| P4 Low Priority       | Виправити після всього іншого                      |

**Команди запуску середовища:**

```bash
# Dev-середовище
docker-compose up -d                              # запустити PostgreSQL (порт 5432, FIX Space_dev)
turbo dev                                         # запустити API + Web

# Unit-тести API (ізольовано, без БД)
pnpm --filter @fixspace/api test                   # unit-тести
pnpm --filter @fixspace/api test -- --coverage     # coverage report

# Integration-тести API (потребують окремого тестового контейнера)
docker-compose up -d db-test                      # тестова БД (порт 5433, FIX Space_test)
cross-env NODE_ENV=test turbo db:migrate:dev       # застосувати міграції до FIX Space_test
pnpm --filter @fixspace/api test:integration       # integration-тести (NODE_ENV=test встановлюється автоматично)

# Frontend component тести
pnpm --filter @fixspace/web test                   # component/unit тести
pnpm --filter @fixspace/web test:watch             # watch mode

# Навантажувальне тестування (Artillery)
pnpm --filter @fixspace/api test:load              # Запуск сценарію навантаження

# E2E UI-тести (Playwright)
pnpm --filter @fixspace/web test:e2e               # Запуск тестів у фоновому режимі (Headless)
pnpm --filter @fixspace/web test:e2e:ui            # Запуск в інтерактивному інтерфейсі (UI Mode)
```
