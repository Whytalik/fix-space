# Системна інфраструктура та регламент розгортання (Deployment Specification)

> Автоматизація конвеєра CI/CD (GitHub Actions, етапи, кешування) описана в [ci-cd.md](ci-cd.md). Цей документ фокусується на інфраструктурній архітектурі.

У цьому документі наведено технічний опис інфраструктурної архітектури та регламентів розгортання платформи FIX Space. Системна інфраструктура має бути побудована на базі гібридної хмарної модели: клієнтський інтерфейс планується розгорнути на глобальній Edge-платформі Vercel, тоді як серверна частина (API) та реляційна база даних повинні функціонувати у контейнеризованому хмарному середовищі Railway.

## Клієнтський рівень: Платформа Vercel та фреймворк Next.js

Клієнтський застосунок (`apps/web`) має бути спроєктовано з використанням гібридних підходів до рендерингу сторінок — статичної генерації (Static Site Generation, SSG) та динамічного рендерингу на стороні сервера (Server-Side Rendering, SSR).

### Оптимізація та конфігурація процесів збирання

Процес складання проєкту в хмарі Vercel повинен бути тісно інтегрований із системою Turborepo. Це дозволить використовувати технологію віддаленого кешування (Remote Caching), що є критично важливим для монорепозиторіїв. Завдяки цьому планується виключити необхідність повторного збирання та тестування незмінних внутрішніх пакетів (таких як `@fixspace/domain` або спільних конфігурацій лінтера).

- **Конвеєр збирання (Pipeline)**: Vercel має здійснювати інсталяцію залежностей через `pnpm install` із залученням глобального кешу пакетів, після чого ініціювати команду `turbo build`.
- **Середовище виконання (Runtime)**: застосунок повинен функціонувати в середовищі Node.js версії 24.

### Конфігурація змінних середовища

Для забезпечення коректного функціонування клієнтського застосунку мають бути налаштовані такі змінні оточення:

- `NEXT_PUBLIC_API_URL` — визначає адресу точки доступу до REST API серверної частини, що використовуватиметься клієнтським браузером для здійснення мережевих запитів.

## Серверний рівень: Контейнеризована інфраструктура Railway та Docker

Серверна частина системи має розгортатися у вигляді ізольованих контейнерів. Це гарантуватиме повну відповідність конфігурацій та поведінки системи в локальному середовищі розробки та у хмарі.

### Багатоетапне збирання контейнерів (Docker Multi-stage Build)

Процес створення контейнера для API-сервісу має бути реалізований за схемою багатоетапного збирання (Multi-stage Build) відповідно до файлу `Dockerfile`:

1. **Етап побудови (Stage 1: Builder)**: має використовуватися повнофункціональний базовий образ `node:24`. У цьому проміжному контейнері повинні встановлюватися всі залежності (включаючи `devDependencies`), генеруватися клієнтський інтерфейс бази даних Prisma Client та проводитися компіляція вихідного коду TypeScript у JavaScript (`nest build`).
2. **Етап виконання (Stage 2: Runner)**: як основа має використовуватися ультралегкий дистрибутив `node:24-alpine`. До цього контейнера повинні копіюватися лише готові скомпільовані файли (`dist`), схеми даних Prisma та мінімальний набір виробничих залежностей (`production dependencies`). Цей підхід дозволить скоротити розмір фінального образу приблизно до 150 МБ та підвищить рівень безпеки, виключаючи з контейнера зайві системні утиліти.

### Безпека та мережеві налаштування

Для захисту серверної частини від поширених загроз та оптимізації мережевого обміну мають бути реалізовані такі заходи:

- **Порти та маршрутизація**: платформа Railway повинна автоматично призначати та передавати динамічну змінну `PORT` всередину контейнера. Сервер NestJS має бути сконфігурований для прослуховування мережевих інтерфейсів за IP-адресою `0.0.0.0` на вказаному порту.
- **Політика крос-доменних запитів (CORS)**: з метою захисту від міжсайтових атак, параметр `CORS_ORIGIN` у налаштуваннях NestJS має бути жорстко обмежений доменним ім'ям клієнтської частини на Vercel. Дозволені методи взаємодії повинні обмежуватися списком `GET, POST, PUT, DELETE, PATCH`.
- **Криптографічний захист автентифікації**: змінна оточення `JWT_SECRET` повинна застосовуватися алгоритмом HS256 для формування цифрового підпису маркерів доступу (JWT). Довжина секретного ключа має становити не менше 256 біт.

## Рівень збереження даних: СКБД PostgreSQL та ORM Prisma

Як основне сховище структурованих даних має застосовуватися керована система керування базами даних PostgreSQL версії 16.

### Інтеграція та конфігурація бази даних

Сховище даних має бути інтегровано із сервером за допомогою сучасних практик доступу:

- **Оптимізація з'єднань (Connection Pooling)**: для раціонального використання ресурсів сервера за умов високої інтенсивності запитів підключення до бази має здійснюватися через механізм пулу з'єднань (Connection Pool). Ліміти активних сесій та час очікування повинні бути жорстко регламентовані у конфігураційному файлі `schema.prisma`.
- **Управління міграціями**: застосування змін до структури бази даних має бути реалізовано за допомогою команди `prisma migrate deploy`, що повинна виконуватися в автоматичному режимі перед безпосереднім стартом сервера (`prestart`). Цей крок гарантуватиме виконання лише нових скриптів міграцій без загрози втрати існуючої інформації.

## Інтеграція поштових сервісів (Resend SDK)

Для забезпечення роботи сервісів автентифікації (верифікація електронної пошти при реєстрації, відновлення доступу) має бути інтегрована система надсилання транзакційних листів.

### Вибір сервісу та логіка роботи середовищ

У системі планується впровадити гібридне підключення поштового провайдера залежно від середовища виконання:

- **Продуктове середовище (Production)**: як основний метод має використовуватися **Resend SDK**. Це забезпечуватиме надійну доставку через HTTP API, виключаючи проблеми з блокуванням SMTP-портів у хмарних середовищах.
- **Локальна розробка (Development)**: за умови відсутності `RESEND_API_KEY`, сервіс повинен автоматично ініціювати тестовий потік на базі **Nodemailer** та провайдера **Ethereal Email** із виведенням посилання на перегляд листа в консоль API.

### Мережева конфігурація та змінні середовища

Для роботи поштового сервісу на платформі Railway необхідно налаштувати такі змінні:

- `RESEND_API_KEY` — основний секретний ключ від Resend (формат `re_...`).
- `MAIL_FROM` — верифікована адреса відправника (наприклад, `FIX Space <noreply@mail.fixspace.app>`).
- `SMTP_HOST / SMTP_PORT / SMTP_USER / SMTP_PASS` — (опціонально) повинні використовуватися як fallback або для розробки, якщо не вказано ключ Resend.

## Інтеграція хмарного сховища зображень (Cloudinary SDK)

Для збереження медіафайлів користувачів (таких як аватари профілю) без зберігання їх локально у тимчасовій файловій системі Docker-контейнерів Railway інтегровано хмарне сховище **Cloudinary**.

### Логіка роботи та оптимізація

- **Динамічне завантаження**: Завантаження зображень відбувається через `StorageService` з використанням Cloudinary Node.js SDK. Файли оптимізуються та зберігаються у хмарі, а в базу даних записується лише безпечна URL-адреса зображення.
- **Очищення ресурсів**: При видаленні або оновленні аватара старий файл автоматично видаляється з хмари Cloudinary за його `public_id`, щоб уникнути накопичення невикористаних медіаресурсів.

### Конфігурація змінних середовища

Для роботи сервісу зображень на платформі Railway необхідно налаштувати такі змінні:

- `CLOUDINARY_CLOUD_NAME` — назва хмарного облікового запису Cloudinary.
- `CLOUDINARY_API_KEY` — публічний ключ доступу API Key.
- `CLOUDINARY_API_SECRET` — приватний секретний ключ API Secret.
- `CLOUDINARY_AVATAR_FOLDER` — (опціонально) назва цільової папки для зберігання аватарів у хмарі (за замовчуванням `fixspace/avatars`).

## Моніторинг, самодіагностика та документування

Поточний контроль працездатності системи та діагностика її стану в реальному часі мають здійснюватися засобами спеціалізованого модуля `@nestjs/terminus`.

1. **Перевірка активності (Liveness Probe)**: уніфікований ендпоінт `/health` повинен відповідати на запити моніторингу для підтвердження того, що server перебуває в робочому стані й здатний приймати вхідні з'єднання.
2. **Перевірка готовності (Readiness Probe)**: має виконувати комплексний аналіз внутрішніх систем, перевіряючи з'єднання з базою даних (`database`) та поточне навантаження на оперативну пам'ять процесу (`memory_heap`).
3. **Документування інтерфейсу (API Documentation)**: інтерактивне середовище Swagger UI має генеруватися в автоматичному режимі на основі декораторів коду NestJS, надаючи актуальну специфікацію інтерфейсу для розробників та зовнішніх інтеграцій.
