## 3.12 Інтеграція із зовнішніми сервісами (Фактична реалізація)

FIX Space підтримує автоматичну фонову синхронізацію торгових даних із зовнішніх торгових платформ для автоматичного наповнення пресетної бази Trading Journal.

---

### 1. Криптовалютні біржі

#### Binance (Реалізовано)

- [x] **Підтримуваний ринок:** Синхронізація підтримується **виключно для USD-M Perpetual Futures**. Spot та COIN-M Futures наразі в коді не реалізовано.
- [x] **Авторизація:** Підключення здійснюється через API Key та API Secret.
- [x] **Суворий аудит безпеки (Read-only Check):** Під час валідації ключів перевіряються права доступу на біржі:
  - Права на читання (`enableReading`) мають бути обов'язково **активні**.
  - Будь-які торгові права (`enableSpotAndMarginTrading`, `enableFutures`, `enableMargin`) та права на виведення (`enableWithdrawals`) мають бути обов'язково **вимкнені**. Якщо хоча б одне з цих прав активоване — підключення відхиляється з помилкою безпеки.
- [x] **Обмеження підключень:** Ліміт до 3 підключень акаунтів Binance на одного користувача.
- [x] **Початкова синхронізація:** При першому підключенні за замовчуванням завантажується торгова історія за останні 30 днів.
- [x] **Робота з лімітами API (Chunking):** Запити до Binance API розбиваються на 7-денні часові інтервали для запобігання помилкам перевищення часового діапазону. Запити виконуються паралельними пакетами (batching) по 5-10 запитів.
- [x] **Визначення активних пар:** Система сканує поточні відкриті позиції користувача та історію нарахування прибутків (income history), щоб виявити торгові пари для синхронізації угод.
- [x] **Агрегація угод:** Окремі виконання (fills) групуються за `orderId`. Після цього ордери із протилежними напрямками послідовно зіставляються (Long/Short на основі `positionSide`) для формування закритих позицій (Trades).
- [x] **Заповнення полів журналу:** Автоматично заповнюються поля: назва пари (символ), напрямок (Long/Short), обсяг позиції, середньозважена ціна входу та виходу, Realized P&L, комісія, фінансування (funding fee), час відкриття та закриття, леверидж та акаунт підключення.
- [x] **Запобігання дублікатам:** Кожна угода отримує унікальний ідентифікатор на основі ключів транзакцій біржі, унеможливлюючи повторне збереження.

---

### 2. Загальна логіка синхронізації та безпека

- [x] **Шифрування Credentials:** Усі токени, API-ключі та секрети шифруються у базі даних за допомогою алгоритму **AES-256-GCM** на рівні бекенду. При поверненні налаштувань клієнту значення ключів маскуються (приховуються).
- [x] **Конфлікт синхронізації:** Синхронізація працює в режимі «тільки створення» (append-only) і ніколи не перезаписує існуючі записи в Trading Journal, зберігаючи всі нотатки та ручні редагування користувача.
- [x] **Синхронізація акаунтів:** Автоматично зв'язує створений запис угоди з обліковим записом у Trading Journal, якщо `Account ID` у журналі збігається з ідентифікатором підключення інтеграції.
- [x] **Помилки синхронізації:** Після 3 послідовних невдалих спроб синхронізації (наприклад, недійсний API-ключ або мережева помилка) система створює для користувача in-app сповіщення типу `WARNING`.
