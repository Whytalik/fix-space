## 3.13 Формули

**Формула** — це автоматично обчислювана властивість лише для читання. Значення перераховується щоразу, коли змінюється будь-яка з властивостей, на яку посилається формула в межах того самого запису. Ключовий принцип: користувач думає результатами, а не механікою — він не пише код і не знає синтаксис. Замість цього він обирає готовий сценарій обчислення та заповнює пропуски через дропдауни.

При додаванні FORMULA-властивості відкривається **галерея named calculations** — картки з реальними задачами в мові трейдера. Якщо жоден пресет не підходить — доступний покроковий конструктор, де користувач також нічого не пише вручну. Весь синтаксис формул (функції, оператори, посилання) генерується системою за лаштунками та ніколи не відображається кінцевому користувачу.

**Загальні обчислення:**

| Назва                 | Що обчислює                                                                                                                                                              | Тип результату                     |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------- |
| Умовний текст         | Показує одну з двох налаштовуваних текстових міток залежно від умови: якщо значення обраного поля відповідає заданій умові — перша мітка, інакше — друга                 | `TEXT`                             |
| Різниця між датами    | Обчислює тривалість між двома DATE-полями; відображається у обраному форматі (години, дні, тижні або HH:mm)                                                              | `DURATION`                         |
| Відсоток              | Одне числове поле поділити на інше та помножити на 100                                                                                                                   | `NUMBER`                           |
| По пов'язаних записах | Агрегувати або перелічити значення обраного поля у пов'язаній базі через RELATION-властивість                                                                            | `NUMBER` · `DATE` · масив об'єктів |
| Середній бал          | Арифметичне середнє від 2 до 5 обраних числових або RATING-полів; використовується для зведеної оцінки якості виконання угоди                                            | `RATING`                           |
| Категорія за порогом  | Присвоює текстову мітку залежно від того, в який числовий діапазон потрапляє обране поле (до 3 порогів і 3 міток); наприклад, A / B / C або Низький / Середній / Високий | `TEXT`                             |

**Обчислення для трейдерів:**

| Назва               | Що обчислює                                                                                                                                                     | Тип результату |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------- |
| R-Multiple          | Ділить значення числового поля P&L на значення числового поля ризику; показує скільки R повернула угода (наприклад, +2.4R або −1.0R)                            | `NUMBER`       |
| Запланований RR     | Обчислює ризик/дохід з цінових рівнів до входу: (Ціна цілі − Ціна входу) / (Ціна входу − Стоп-лосс); показує скільки R заплановано до відкриття угоди           | `NUMBER`       |
| Ризик % від балансу | (Сума ризику / Баланс акаунту) × 100; ключова метрика для prop firm трейдерів — більшість компаній обмежують ризик на угоду до 1–2% від депозиту                | `NUMBER`       |
| Дотримання правил   | Відсоток дотриманих правил угоди: ділить кількість відмічених CHECKBOX-полів на загальну кількість обраних CHECKBOX-полів; 100% означає що всі правила виконані | `PROGRESS`     |

**Функціональні вимоги:**

Галерея та вибір обчислення:

- 📝 Додано до Issue (#119) При додаванні FORMULA-властивості відкривається галерея пресетних обчислень із назвами в мові трейдера (не технічні терміни)
- 📝 Додано до Issue (#119) Кожен пресет містить короткий опис того, що буде відображатись у колонці
- 📝 Додано до Issue (#119) Наявна опція «Свій розрахунок» для побудови власного обчислення через покроковий конструктор
- 📝 Додано до Issue (#119) Вибір пресету одразу переводить до кроку заповнення пропусків

Заповнення пропусків (fill-in-the-blanks):

- 📝 Додано до Issue (#119) Кожне поле пресету або конструктора заповнюється виключно через дропдауни: вибір властивості, оператора або фіксованого значення
- 📝 Додано до Issue (#119) Користувач не вводить жодного синтаксису, функцій або посилань вручну
- 📝 Додано до Issue (#119) Доступний для вибору лише властивості сумісного типу даних (число → числові поля, дата → поля типу DATE тощо)
- 📝 Додано до Issue (#119) FORMULA-властивості не відображаються у списку доступних полів для вибору — це унеможливлює циклічні залежності на рівні інтерфейсу
- 📝 Додано до Issue (#119) Вихідний тип (`NUMBER`, `TEXT`, `DATE`, `BOOLEAN`, `RATING`, `PROGRESS`, `DURATION`, масив об'єктів) визначається автоматично з обраного сценарію та не потребує явного вибору

Покроковий конструктор (для «Свого розрахунку»):

- 📝 Додано до Issue (#119) Підтримувані операції: математичні (+, −, ×, ÷), порівняння (більше, менше, дорівнює, не дорівнює), різниця між датами, об'єднання тексту
- 📝 Додано до Issue (#119) Конструктор обмежений двома рівнями: `[поле / значення] [операція] [поле / значення]`; для умовного відображення: `якщо [умова] → показати [текст], інакше [текст]`

Пресет «Запланований RR»:

- 📝 Додано до Issue (#119) Доступний лише якщо в базі є мінімум три числових поля (Ціна входу, Стоп-лосс, Ціна цілі)
- 📝 Додано до Issue (#119) Wizard із трьох дропдаунів: «Ціна входу», «Стоп-лосс», «Ціна цілі» — кожен вибирається з NUMBER-полів бази
- 📝 Додано до Issue (#119) Формула за лаштунками: `(Ціль − Вхід) / (Вхід − Стоп)` для Long; окрема кнопка «Угода є Short» інвертує знаменник: `(Вхід − Ціль) / (Стоп − Вхід)`
- 📝 Додано до Issue (#119) Вихідний тип — число; від'ємне значення сигналізує про помилку у цінових рівнях (ціль і стоп розміщені неправильно)

Пресет «Дотримання правил»:

- 📝 Додано до Issue (#119) Wizard показує всі CHECKBOX-поля бази у вигляді списку з мультивибором — користувач позначає лише ті, що є правилами угоди
- 📝 Додано до Issue (#119) Система рахує частку відмічених полів від обраних: `(Кількість відмічених / Загальна кількість обраних) × 100`
- 📝 Додано до Issue (#119) Поля, не включені до списку правил, не впливають на результат
- 📝 Додано до Issue (#119) Вихідний тип — число (відсоток від 0 до 100); порожній запис без жодного заповненого CHECKBOX дає 0%

Пресет «Середній бал»:

- 📝 Додано до Issue (#119) Користувач обирає від 2 до 5 полів типу NUMBER або RATING через мультивибір
- 📝 Додано до Issue (#119) Система обчислює арифметичне середнє обраних полів у межах одного запису
- 📝 Додано до Issue (#119) Якщо будь-яке з обраних полів порожнє у записі — воно виключається з обчислення (середнє рахується по заповнених полях)

Пресет «Категорія за порогом»:

- 📝 Додано до Issue (#119) Користувач обирає одне числове поле та задає до 3 правил вигляду «якщо [поле] більше ніж [N] → показати [мітку]»
- 📝 Додано до Issue (#119) Правила перевіряються зверху вниз; перше виконане — застосовується
- 📝 Додано до Issue (#119) Обов'язкове поле «Інакше → показати [мітку]» визначає значення за замовчуванням
- 📝 Додано до Issue (#119) Приклад: Grade > 4 → «Відмінно», Grade > 2 → «Нормально», інакше → «Слабо»

Пресет «По пов'язаних записах»:

- 📝 Додано до Issue (#119) Доступний лише якщо база даних має щонайменше одну RELATION-властивість
- 📝 Додано до Issue (#119) Wizard із трьох кроків: 1) вибір RELATION-властивості; 2) вибір поля у пов'язаній базі; 3) вибір операції агрегації
- 📝 Додано до Issue (#119) Підтримувані операції: Кількість (будь-яке поле) → `NUMBER`; Сума / Середнє / Мін / Макс (лише числові поля) → `NUMBER`; Найраніша / Найпізніша (лише DATE-поля) → `DATE`; Список значень (будь-яке поле) → масив об'єктів — повертає перелік усіх значень обраного поля з пов'язаних записів
- 📝 Додано до Issue (#119) Доступні поля пов'язаної бази фільтруються за сумісністю з обраною операцією — несумісні поля у дропдауні не відображаються
- 📝 Додано до Issue (#119) Значення перераховується автоматично при зміні будь-якого пов'язаного запису

Попередній перегляд:

- 📝 Додано до Issue (#119) Панель налаштування формули відображає обчислене значення для першого наявного запису у реальному часі
- 📝 Додано до Issue (#119) При відсутності записів у базі — відображається приклад з підставленими тестовими значеннями
- 📝 Додано до Issue (#119) Попередній перегляд оновлюється миттєво при зміні будь-якого параметру конструктора

Поведінка у базі даних:

- 📝 Додано до Issue (#119) Значення FORMULA-властивості є лише для читання — редагувати вручну неможливо
- 📝 Додано до Issue (#119) Перерахунок відбувається автоматично при збереженні будь-якої зміни в записі
- 📝 Додано до Issue (#172) FORMULA-властивість доступна для фільтрації, сортування та групування у поданнях
- 📝 Додано до Issue (#119) Стандартні пресети та конструктор посилаються лише на властивості того ж запису; доступ до пов'язаних баз даних реалізований виключно через пресет «По пов'язаних записах»
