## 3.14 Автоматизація

**Автоматизація** — це налаштування правил без коду (no-code rules) виду «КОЛИ→ТО» усередині бази даних. Вони виконують повторювані рутинні операції автоматично при настані певних подій: зміні значень властивостей, додаванні нового запису або за часовим розкладом.

Для трейдера автоматизація спрощує підтримку чистоти та структури журналу: коли угода закривається — автоматично створити Mistake запис за певних умов, щотижня — створювати звіт, при створенні огляду — підтягнути всі угоди тижня.

**Загальні шаблони:**

| Шаблон                       | Сценарій                                                                                           |
| ---------------------------- | -------------------------------------------------------------------------------------------------- |
| Поставити дату при зміні     | Коли обране поле змінюється на певне значення — автоматично заповнити DATE-поле поточною датою     |
| Заповнити поле при створенні | Коли створюється новий запис — встановити одне або кілька полів у задані значення за замовчуванням |
| Позначити для огляду         | Коли умова виконується — встановити CHECKBOX-поле у значення «відмічено»                           |
| Нагадати за розкладом        | У визначений день і час — створити новий запис у обраній базі даних                                |
| Скопіювати в іншу базу       | Коли умова виконується — створити запис в іншій базі даних із перенесенням обраних полів           |

**Шаблони для трейдерів:**

| Шаблон                              | Сценарій                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Автологування помилки               | Коли поле «Rule Violation» (CHECKBOX) стає відміченим (true) — автоматично створити запис у базі Mistakes із назвою угоди та датою; виключає ручне дублювання між базами                                                                                                                                                                                                                                |
| Тижневий Performance Review         | Щопонеділка о вибраний час — створити запис у Performance Review із Period Type = Weekly та датами поточного тижня; нагадує почати ретроспективу                                                                                                                                                                                                                                                        |
| Місячний Performance Review         | У перший день кожного місяця о вибраний час — створити запис у Performance Review із Period Type = Monthly та датами попереднього місяця                                                                                                                                                                                                                                                                |
| Квартальний Performance Review      | У перший день кожного кварталу (1 січня, 1 квітня, 1 липня, 1 жовтня) о вибраний час — створити запис у Performance Review із Period Type = Quarterly та датами попереднього кварталу                                                                                                                                                                                                                   |
| Авто-категоризація результату       | Коли поле P&L змінюється на значення більше 0 — встановити SELECT-поле «Результат» = «Прибуткова»; аналогічне окреме правило для від'ємних значень; усуває ручну категоризацію кожної угоди                                                                                                                                                                                                             |
| Позначити перевищення ліміту ризику | Коли поле «Risk ($)» перевищує заданий поріг у доларах — встановити CHECKBOX «Перевищено ліміт» = відмічено; нагадує переглянути розмір позиції перед наступною угодою; поле FORMULA «Risk %» доступне для перегляду, але не може бути тригером автоматизації (FORMULA-поля виключені — див. опис блоку «КОЛИ»)                                                                                         |
| Авто-зв'язок угод з оглядом         | Коли запис у Performance Review створено — знайти всі записи Trading Journal, де `Date` потрапляє в діапазон `Date From`..`Date To`, та заповнити поле `Related Trades`; другий варіант тієї самої автоматизації (окрема): тригер «`Date To` заповнюється» — охоплює записи, створені вручну, де дати вводяться після створення; якщо `Date From` або `Date To` порожні — дія пропускається автоматично |
| Авто-зв'язок помилок з оглядом      | Коли запис у Performance Review створено або поле `Date To` заповнюється — знайти всі записи Mistakes, де хоча б одна пов'язана рутина (`Routines.Date`) потрапляє в діапазон `Date From`..`Date To`, та заповнити поле `Mistakes`; разом із «Авто-зв'язок угод» забезпечує повну картину тижня / місяця / кварталу без ручного збирання                                                                |
| Авто-зв'язок угод з рутиною         | Коли поле `Date` заповнюється у Daily Routine — знайти всі записи Trading Journal із тією самою датою та прив'язати їх до поля `Trades`; трейдер відкриває рутину і відразу бачить угоди дня без ручного перелінкування                                                                                                                                                                                 |

**Функціональні вимоги:**

Галерея та вибір шаблону:

- [x] При створенні автоматизації відкривається галерея шаблонів із назвами та коротким описом сценарію кожного
- [x] Кожен шаблон містить структуру «КОЛИ → ТО» з підказками, які саме пропуски потрібно заповнити
- [x] Наявна опція «Почати з нуля» — відкриває порожню структуру «КОЛИ → ТО» з тими самими дропдаунами
- [x] Назви «тригер», «умова», «дія» не використовуються в інтерфейсі

Блок «КОЛИ» (Trigger):

- [x] **Поле змінюється** — вибір конкретної властивості та, за необхідності, умови на нове значення: конкретне значення (наприклад, «Статус» змінюється на «Reviewed»); або для полів без переліку значень (DATE, NUMBER, TEXT, RELATION) — «заповнюється» (значення з порожнього стає непорожнім) або «очищається»; поля типу FORMULA виключені з цього тригера — їхнє значення перераховується системою автоматично і не є «зміною» з точки зору автоматизації
- [x] **Запис створено** — спрацьовує при додаванні будь-якого нового запису до цієї бази даних
- [x] **За розкладом** — вибір інтервалу (щодня / щотижня / щомісяця), дня тижня або числа місяця та часу доби; мінімальний інтервал — 1 день

Блок «ТО» (Action):

- [x] **Встановити значення поля** — вибір властивості поточного запису та нового значення: фіксоване значення, поточна дата або значення іншої властивості цього запису
- [x] **Створити запис** — вибір цільової бази даних (та сама або інша в межах простору), перелік полів нового запису та їхніх значень; значення полів можуть бути фіксованими або перенесеними з полів запису, що спричинив тригер
- [x] **Зв'язати записи** — пошук записів у вибраній базі даних за заданими фільтрами та додавання знайдених записів до RELATION-властивості поточного запису; фільтри підтримують посилання на поля поточного запису (наприклад, `Date between Date From..Date To`); режим запису: `replace` (замінити всі наявні зв'язки) або `append` (додати до існуючих); якщо будь-яке з полів, задіяних у фільтрах, порожнє — дія пропускається без помилки
- [x] До однієї автоматизації можна додати до 5 дій; дії виконуються послідовно зверху вниз

Тестування автоматизації:

- [x] Кнопка «Перевірити» запускає автоматизацію вручну на обраному існуючому записі без збереження результату
- [x] Після тесту відображається результат у людській мові: які поля були б змінені та якими значеннями
- [x] Тестовий запуск доступний до активації автоматизації

Управління автоматизаціями:

- [x] Перемикач активна / неактивна для кожної автоматизації (вимкнена автоматизація не спрацьовує, але зберігається)
- [x] Редагування будь-якого параметру автоматизації без необхідності видаляти та створювати заново
- [x] Видалення автоматизації (з підтвердженням)
- [x] Ліміт: до 10 автоматизацій на базу даних

Журнал запусків (Logs):

- [x] Для кожної автоматизації доступний журнал останніх 50 запусків
- [x] Кожен рядок журналу містить: дату та час, назву запису, що спричинив тригер, результат у людській мові («встановлено "Reviewed At" = 25.05.2025») або опис помилки («пропущено: поле P&L порожнє»)
- [x] Помилка одного запуску не блокує наступні

Захист від рекурсії та помилок:

- [x] Якщо дія автоматизації змінює поле, яке є тригером цієї ж автоматизації — новий тригер не запускається (захист від рекурсії)
- [x] Автоматизації різних баз даних не взаємодіють між собою каскадно: якщо дія автоматизації створює запис в іншій базі — автоматизації тієї бази не запускаються через цей запис; це виключає можливість циклів типу A → B → A між різними базами, але не знімає з користувача відповідальність за логічну суперечність усередині однієї бази (кілька автоматизацій, що змінюють одне й те саме поле одна одній)
- [x] Автоматизація з тригером «За розкладом» виконується лише якщо відповідна база даних існує та доступна
