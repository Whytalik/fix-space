## #166 [Integration] Connect Binance/Bybit/OKX with auto-sync and manual trigger

**State:** OPEN
**Milestone:** Full Release
**Labels:** `feature`, `mod:integration`, `layer:api`, `layer:db`, `priority:could`

### User Story

**US-091 / US-092 / US-093:** Як крипто-трейдер, я хочу підключити свої біржові акаунти (Binance, Bybit, OKX), щоб мої угоди автоматично з'являлися в журналі.

### Description

Інтеграція з криптовалютними біржами через API-ключі (read-only). Система автоматично завантажує закриті позиції (Futures/Spot) кожні 5 хвилин та мапить їх у властивості Trading Journal. Підтримується ручний запуск синхронізації.

### Acceptance Criteria

- [ ] Підключення через API Key + Secret; система відхиляє ключі з правами на торгівлю або виведення
- [ ] Вибір ринку (USDT-M, COIN-M, Spot) при налаштуванні
- [ ] Автоматична синхронізація кожні 5 хвилин у фоновому режимі
- [ ] Кнопка «Sync Now» для миттєвого оновлення
- [ ] Завантаження історії за останні 30–90 днів при першому підключенні
- [ ] Ігнорування дублікатів за унікальним Position/Trade ID біржі
- [ ] Автоматичне заповнення властивості `Account` на основі підключення

**API:**

- `POST /api/integrations/crypto/connect` — body: `{ provider, apiKey, apiSecret, config }`

- Бібліотека: `ccxt` для уніфікованого доступу до бірж

**DB:**

- [ ] Сервіси для кожної біржі (Binance, Bybit, OKX)
- [ ] Encrypted storage для API-ключів
- [ ] Background sync worker з ігноруванням дублікатів
- [ ] Маппінг полів біржі у схему Trading Journal

**Frontend (Web):**

- [ ] Форма підключення біржі (API Key, Secret, Marketplace selection)
- [ ] Індикатор статусу синхронізації

### Functional Requirements

- [ ] **[§3.12]** Підключення біржі через API Key + Secret (read-only)
- [ ] **[§3.12]** Перевірка прав ключа при підключенні: якщо ключ має права на торгівлю або виведення коштів — підключення відхиляється з відповідним повідомленням
- [ ] **[§3.12]** Вибір типу ринку для синхронізації (Futures / Spot або обидва)
- [ ] **[§3.12]** Автоматична синхронізація нових закритих угод з інтервалом 5 хвилин
- [ ] **[§3.12]** Ручний запуск синхронізації на вимогу
- [ ] **[§3.12]** Завантаження торгової історії за вибраний діапазон дат при першому підключенні (глибина залежить від біржі: до 2 років для Bybit V5; для Binance — до 30 днів через API, решта — через CSV-імпорт)
- [ ] **[§3.12]** Зіставлення полів біржі з властивостями бази Trading Journal: символ, напрямок (Long / Short), обсяг, ціна входу/виходу, P&L, комісія, funding rate (для perpetual), час відкриття та закриття, тип ринку, леверидж
- [ ] **[§3.12]** Ігнорування дублікатів при повторній синхронізації (ідентифікація за унікальним position ID біржі)
- [ ] **[§3.12]** Автоматичне заповнення поля `Account` у синхронізованому записі Trading Journal — підбирається запис Accounts, де поле `Account ID` відповідає ідентифікатору підключеного акаунту інтеграції; якщо збіг не знайдений — поле залишається порожнім
- [ ] **[§3.12]** Відображення статусу підключення: активне / помилка автентифікації / відключене
- [ ] **[§3.12]** Відключення інтеграції (збережені записи залишаються в журналі)
- [ ] **[§3.12]** Ліміт: до 3 підключень на кожну біржу на акаунт; кожне підключення отримує довільну мітку-назву (наприклад, «Binance Main», «Binance Prop»)
- [ ] **[§3.12]** **Одиниця синхронізації — позиція:** система синхронізує закриті позиції, а не окремі часткові виконання (fills). Одна закрита позиція (від відкриття до повного закриття) = один запис у базі Trading Journal. Ціна входу та виходу розраховуються як середньозважені по всіх fills позиції. Для Binance та Bybit використовується Position History API, який надає агреговані дані; для MT5 — групування по position ID.
- [ ] **[§3.12]** **Конфлікт синхронізації та редагування:** синхронізація лише **створює нові записи** — вона ніколи не оновлює та не перезаписує записи, які вже існують у базі даних. Якщо запис з відповідним position ID вже є в журналі (незалежно від того, чи редагував його користувач), повторна синхронізація його ігнорує. Це гарантує збереження всіх нотаток, тегів, значень властивостей і контентної області, доданих вручну.
- [ ] **[§3.12]** **Маппінг полів:** маппінг між полями брокера і властивостями бази Trading Journal є фіксованим і прив'язаний до внутрішніх ідентифікаторів системних властивостей (не до їхніх назв). Перейменування системної властивості користувачем не порушує синхронізацію. Поля брокера, що не мають відповідної системної властивості, не імпортуються.
- [ ] **[§3.12]** **Funding rate (для perpetual futures):** funding rate за весь час утримання позиції вже включений у значення Realized P&L, яке повертає Position History API Binance та Bybit. Окремого запису для funding rate не створюється — він є частиною підсумкового P&L угоди.
- [ ] **[§3.12]** **Валюта:** числові значення (P&L, комісія, обсяг) зберігаються у валюті брокерського акаунту як є (USDT для Binance/Bybit USDT-M, USD або EUR для MT5 залежно від акаунту). Конвертація валют не виконується. Валюта акаунту відображається поруч із числовими значеннями в картці запису.
- [ ] **[§3.12]** **Початкова синхронізація:** при першому підключенні користувач вибирає діапазон дат для завантаження торгової історії. Значення за замовчуванням — останні 90 днів. Максимальна глибина залежатиме від обмежень API: Bybit — до 2 років, Binance — до 30 днів (решту завантажити через CSV-імпорт), MT5 та cTrader — необмежено (залежить від брокера).
- [ ] **[§3.12]** **Мітки джерела:** синхронізовані записи позначаються міткою підключення (задається користувачем при налаштуванні, наприклад «Binance Main», «IC Markets cTrader», «MT5 Prop»); мітка відображається в картці запису та доступна для фільтрації.
- [ ] **[§3.12]** **Безпека:** Credentials (API-ключі, паролі, OAuth-токени) зберігаються у зашифрованому вигляді; після збереження відображаються лише у замаскованому вигляді.
- [ ] **[§3.12]** Всі синхронізовані записи є повноцінними записами бази даних — їх можна редагувати, доповнювати нотатками та контентною областю, видаляти
- [ ] **[§3.12]** Видалення запису у FIX Space не впливає на дані на біржі або в терміналі

### References

- Functional: §3.12 Інтеграція із зовнішніми сервісами

## #169 [Integration] In-app alert on 3 consecutive sync failures

**State:** OPEN
**Milestone:** Full Release
**Labels:** `feature`, `mod:integration`, `layer:api`, `layer:web`, `priority:could`

### User Story

**US-096:** Як користувач, я хочу знати, якщо автоматична синхронізація з моєю біржею або брокером не працює, щоб я міг виправити проблему.

### Description

Система моніторингу статусу інтеграцій. Якщо синхронізація з певним джерелом (Binance, MT5 тощо) зазнає невдачі 3 рази поспіль (напр. через зміну пароля або невалідний API-ключ), користувач отримує in-app сповіщення з описом проблеми.

### Acceptance Criteria

- [ ] Трекінг послідовних помилок для кожного підключення
- [ ] При досягненні 3 помилок — створення сповіщення (type: Error)
