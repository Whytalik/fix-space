# Нефункціональні вимоги

Визначають якісні характеристики системи, що забезпечують її ефективну та надійну роботу в умовах щоденного використання. За основу взято модель ISO/IEC 25010.

---

## Продуктивність

| Операція                                                      | Цільовий час           |
| ------------------------------------------------------------- | ---------------------- |
| Відкриття робочого простору                                   | ≤ 300 мс               |
| Відкриття бази даних, перехід до запису, застосування фільтра | ≤ 300 мс               |
| Створення запису (відгук інтерфейсу)                          | без відчутних затримок |
| Перерахунок FORMULA-властивостей після зміни залежного поля   | ≤ 1 с                  |
| Завантаження сторінки Статистики (графіки)                    | ≤ 2 с                  |
| Глобальний пошук (перші результати)                           | ≤ 300 мс               |

- Фільтрація, сортування та групування виконуються в режимі реального часу без перезавантаження сторінки.
- Виконання автоматизацій відбувається у фоновому режимі — без впливу на швидкість інтерфейсу.
- Продуктивність фільтрації та пошуку не деградує при зростанні обсягу записів у базі даних.

---

## Надійність

- Система гарантує збереження даних користувача під час технічних збоїв.
- У разі неможливості збереження — користувач отримує явне попередження.
- Формули та автоматизації є детермінованими: однакові вхідні дані завжди повертають однаковий результат.
- Синхронізація з інтеграціями не перезаписує вже існуючі записи — лише створює нові.
- Усі транзакції бази даних відповідають вимогам ACID (через `prisma.$transaction()`).

---

## Відмовостійкість

- Ініціалізація робочого простору (4-pass) виконується в єдиній транзакції — при помилці на будь-якому кроці відбувається повний rollback.
- Контентна область запису автоматично зберігається кожні 5 хвилин та при значних змінах (макс. 50 знімків на запис).
- Cycle detection для FORMULA-властивостей: виявляє циклічні залежності при збереженні конфігурації та під час виконання.
- Recursion guard для автоматизацій: зміна поля автоматизацією не тригерить ту саму автоматизацію повторно; каскадне спрацювання між базами даних заблоковано.
- Механізм повторних спроб (retry): TanStack Query — до 3 повторів при мережевій помилці; синхронізація інтеграцій — після 3 послідовних помилок формується сповіщення, спроби продовжуються кожні 30 хвилин.

---

## Безпека

- Дані кожного користувача повністю ізольовані на рівні робочого простору (resource-level ownership).
- Доступ до будь-яких ресурсів — виключно в межах автентифікованої сесії.
- Паролі зберігаються у вигляді хешів (bcrypt, salt rounds = 10).
- JWT-автентифікація через HttpOnly cookies: access-токен діє 15 хвилин, refresh-токен — 30 днів.
- Підтримка входу через Google OAuth 2.0 (Authorization Code Flow) — без передачі пароля системі.
- При зміні пароля всі активні сесії анулюються автоматично.
- API-ключі та облікові дані інтеграцій зберігаються у зашифрованому вигляді; після збереження відображаються лише в замаскованому вигляді.
- Ключ API-інтеграції з правами на торгівлю або виведення коштів — відхиляється при підключенні.
- Rate limiting на ендпоінті входу: не більше 5 спроб за 60 секунд з однієї IP-адреси (ThrottlerGuard).
- Глобальний rate limiting: не більше 200 запитів за хвилину на всі API endpoints.
- CORS: дозволено лише домен клієнтської частини зі змінної `CORS_ORIGIN`; `credentials: true`; методи: GET, POST, PATCH, DELETE, OPTIONS.
- Валідація вхідних даних: NestJS ValidationPipe (whitelist — відкидання незнайомих полів, forbidNonWhitelisted — помилка при зайвих полях, transform — автоматична конвертація типів).
- Управління секретами: змінні оточення в `.env.*` файлах (не версіонуються); шаблон `.env.example` — у git.
- Дотримання вимог GDPR та ePrivacy Directive: наявність інформаційного банера про використання файлів cookie та отримання згоди користувача.
- Свідомо не реалізовано в v1.0: шифрування `PropertyValue.value` на рівні застосунку (компенсується encryption at-rest інфраструктури), двофакторна автентифікація (2FA), системний security audit log.

---

## Зручність використання

- Будь-яка типова операція щоденного сценарію виконується не більше ніж за 3 кліки від поточного місця в інтерфейсі.
- Повідомлення про помилки містять зрозумілий опис причини та рекомендацію щодо подальших дій — без технічних кодів.
- Перед незворотними операціями (видалення) відображається діалог підтвердження.
- Глобальний пошук відкривається з будь-якої сторінки через Ctrl+K / Cmd+K.

---

## Доступність

- Коректна робота в останніх двох версіях браузерів: Chrome, Firefox, Edge, Safari.
- Для функціонування системи необхідне стабільне підключення до інтернету; офлайн-режим не підтримується.
- Інтерфейс доступний двома мовами: українська та англійська.

---

## Масштабованість

- Серверна архітектура побудована за принципом модульного моноліту — новий функціонал додається без модифікації існуючого коду.
- Продуктивність фільтрації та пошуку не деградує зі зростанням кількості записів.
- Стабільність зв'язків між сутностями зберігається при великому обсязі пов'язаних даних.

---

## Спостережуваність

- Ендпоінт `/health`: liveness probe (сервер працює) та readiness probe (стан БД, навантаження на пам'ять) через `@nestjs/terminus`.
- Інтерактивна документація API через Swagger UI — генерується автоматично з декораторів NestJS.
- Звіти про покриття коду тестами завантажуються в Codecov при кожному CI-запуску.
- Артефакти CI: звіти покриття (7 днів), Playwright-звіти (7 днів), артефакти збірки (1 день).

---

## Інфраструктура та розгортання

- **Frontend:** Next.js на Vercel Edge CDN — SSG, SSR, Streaming з Suspense; автоматичний preview для кожного PR.
- **Backend:** NestJS у Docker-контейнерах Railway (multi-stage build: `node:24` → `node:24-alpine`, ~150 МБ).
- **Database:** PostgreSQL 16 на Railway з connection pooling для уникнення ліміту підключень.
- **Медіа:** Cloudinary — зображення оптимізуються при завантаженні, автоматично видаляються при оновленні аватара.
- **Email:** Resend SDK (production) / Nodemailer + Ethereal Email (development) — автоматичний вибір за наявністю `RESEND_API_KEY`.
- **CI/CD:** GitHub Actions — pipeline: lint → test (coverage) → security audit → integration test → build → docker build; кешування: pnpm store, Turborepo (.turbo), розумна інвалідація кешу.
- **Середовища:** Staging (develop, автоматичний деплой через `railway up`) → Production (main, через PR після проходження CI).
- **Zero-downtime deployment:** Railway гарантує безперебійне оновлення контейнерів API.
- **Локальна розробка:** docker-compose (PostgreSQL), `turbo dev` (API + Web), окремий тестовий контейнер `db-test` (порт 5433).
