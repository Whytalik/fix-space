# @nucleus/database

Prisma ORM configuration and client for the Nucleus monorepo. Provides a singleton
`PrismaClient` instance backed by a PostgreSQL connection pool via the `@prisma/adapter-pg`
driver adapter, and re-exports all Prisma-generated types.

## Structure

```
packages/database/
├── prisma/
│   ├── schema.prisma          # Database schema — models, enums, relations
│   ├── prisma.config.ts       # Prisma CLI configuration (datasource, output path)
│   └── migrations/            # Migration files generated by prisma migrate dev
├── generated/
│   └── client/                # Auto-generated Prisma Client (git-ignored, rebuild with db:generate)
└── src/
    └── client.ts              # PrismaClient singleton with env loading and adapter setup
```

## Prisma setup

### Generator

Uses the Prisma 7 `prisma-client` generator, which produces a Rust-free client
with faster queries and a smaller bundle than the legacy `@prisma/client` approach:

```prisma
generator client {
  provider = "prisma-client"
  output   = "../generated/client"
}
```

The generated client is output to `packages/database/generated/client/` and is
**git-ignored** — it must be regenerated after any schema change.

## Client

### Singleton

`src/client.ts` exports a singleton `prisma` instance that prevents multiple
connection pools from being created during hot-reload in development:

```ts
import { prisma } from "@nucleus/database";

const users = await prisma.user.findMany();
```

The singleton pattern guards on `global.prisma` — in production (`NODE_ENV=production`)
the guard is skipped and a fresh client is created per process.

### Driver adapter

The client uses `@prisma/adapter-pg` with a `pg` connection pool rather than
Prisma's default TCP driver. This is required in Prisma 7 when using certain
driver-level features and is the recommended approach for PostgreSQL:

```ts
const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const adapter = new PrismaPg(pool);
export const prisma = new PrismaClient({ adapter });
```

### Environment loading

`client.ts` loads environment variables at import time using `dotenv`, resolving
paths relative to the monorepo root:

1. `.env.{NODE_ENV}` (e.g. `.env.development`, `.env.production`)
2. `.env` — fallback

This means importing `@nucleus/database` in any context will automatically
load the correct env file without any additional setup in the consuming app.

### Re-exports

`client.ts` re-exports everything from the generated Prisma client:

```ts
export * from "../generated/client/client.js";
```

This means `@nucleus/database` is the single import point for both the `prisma`
instance and all generated types (`Prisma`, model types, enums, etc.).

## Commands

All commands use `dotenv-cli` to load env vars from `.env.development` and `.env`:

```bash
# Regenerate Prisma Client after schema changes (required before dev/build)
pnpm --filter @nucleus/database db:generate

# Create and apply a new migration (development only)
pnpm --filter @nucleus/database db:migrate:dev

# Apply pending migrations without creating new ones (production)
pnpm --filter @nucleus/database db:migrate:deploy

# Push schema changes without a migration file (prototyping only)
pnpm --filter @nucleus/database db:push

# Open Prisma Studio (visual database browser)
pnpm --filter @nucleus/database studio

# Format schema.prisma
pnpm --filter @nucleus/database format

# Seed the database
pnpm --filter @nucleus/database db:seed
```

Or via the root monorepo shortcuts:

```bash
turbo db:generate
turbo db:migrate:dev
```

## Environment

```
DATABASE_URL=postgresql://user:password@localhost:5432/nucleus?schema=public
```

Required in `.env.development` (local) or as an environment variable in production.
The `?schema=public` suffix is required — Prisma uses it to scope all queries
to the `public` schema.

## Prisma 7 notes

- **ESM only** — `package.json` has `"type": "module"`; all imports must use `.js` extensions
- **Driver adapter required** — `PrismaClient` must receive an `adapter` option; direct TCP
  connections without an adapter are not supported in this configuration
- **`prisma generate` is explicit** — it no longer runs automatically after `migrate dev`;
  always run `db:generate` after schema changes
- **Generated output is not in `node_modules`** — the client lives in
  `packages/database/generated/client/` and is git-ignored
